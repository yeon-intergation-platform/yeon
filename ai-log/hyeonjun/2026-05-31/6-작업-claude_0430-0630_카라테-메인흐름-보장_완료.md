## 핵심 제약과 해결 (Java 25 + Karate)

- 인프로세스 `@SpringBootTest` + `karate-junit5` 방식은 **Java 25에서 무한 대기(hang)**. jstack 결과 `Suite.run` join 에서 영구 블록(CPU 거의 0), Karate 1.4.1의 GraalJS가 Java 25 미지원.
- Maven Central 최신 Karate는 **1.4.1**(1.5.x 없음).
- 해결: **앱은 Java 25(toolchain)로 띄우고, Karate는 Java 21 standalone(fat jar)으로 그 앱을 HTTP로 친다.** Java 21에서 GraalJS 정상 동작 확인. → `build.gradle`에 karate 의존성을 넣지 않아 `./gradlew test`(Java 25)는 영향 없음.

## 인증 모델 (feature 작성 근거)

- community-chat: permitAll(게스트, 토큰 불필요)
- card-deck-assets / typing-decks(목록) / card-rooms / star-lobby: `X-Yeon-Internal-Token`(ROLE_INTERNAL)으로 충분
- card-decks(쓰기) / typing-decks(쓰기): `X-Yeon-Internal-Token` + `X-Yeon-User-Id` 필요. `card_decks.owner_user_id`가 `public.users` FK라 **쓰기 흐름은 테스트 사용자 시드가 필요**(러너가 자동 시드).
- card-rooms 플레이(start/result/reveal/next): `X-Yeon-Participant-Id` 필요(입장 응답의 participant id).

## 잔여 / 메모

- 실행에 Java 21(Karate) + Java 25(앱) 둘 다 필요. 러너는 `/usr/libexec/java_home -v 21` 로 21 탐지, 25는 bootRun toolchain.
- CI 통합 시: 백엔드 컨테이너 기동 → `run-flows.sh`(KARATE_BASE_URL 지정) 단계로 추가 가능. 인프로세스 통합은 Karate가 Java 25 지원하기 전까지 보류.
- 카운슬링/스페이스/시트 등 CRM 관리 표면은 별도 제품(상태/OAuth 의존 큼)이라 이번 메인 제품 흐름 범위에서 제외.
- 기존 Testcontainers 통합 테스트(`CardDeckAssetUploadIntegrationTest`, 인프로세스 Java 25)는 그대로 유지 — 이미지 멀티파트/%2F 회귀를 `./gradlew test`에서도 보장.
