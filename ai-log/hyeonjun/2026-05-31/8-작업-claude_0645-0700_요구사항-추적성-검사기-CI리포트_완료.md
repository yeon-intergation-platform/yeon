- 작업자: claude (Opus 4.8)

- 브랜치: `chore/requirements-traceability` (base: `main`)
- 요청: requirements.yaml ↔ @REQ 태그 추적성 검사기 + Karate/JaCoCo/REST Docs 리포트 CI artifact 노출

## ① 요구사항 원장 + @REQ 태그

- `apps/backend/requirements.yaml`: 요구사항 7건(ID·title·priority·tests[type,path]). P0 3건(카드에셋·커뮤니티채팅·카드덱), P1 3건, P2 1건.
- 7개 Karate feature 에 `@REQ-*` 태그 부착(Feature 레벨). 예: `@cardDeckAssets @REQ-CARD-ASSET-001`.

## ② 추적성 검사기 (RequirementsTraceabilityTest, JUnit)

`./gradlew test` 에서 빌드로 강제(파일 IO + snakeyaml, Spring 컨텍스트 불필요). 4개 검사:

- (a/c) feature 의 `@REQ` 태그가 원장에 존재 — 삭제된 ID 참조 시 실패
- (b) P0 요구사항인데 테스트 없는 항목 없음
- (d) 원장의 테스트 파일 경로 실존
- (e) karate 로 연결된 요구사항은 해당 feature 에 `@REQ` 태그 실재(양방향)

**음성 테스트로 검증**: 존재하지 않는 `@REQ-FAKE-999`를 feature 에 주입 → BUILD FAILED(검사기가 실제로 잡음). 원복 후 통과.

## ③ CI 테스트 게이트 + 리포트 (`.github/workflows/backend-tests.yml`)

> 기존 백엔드 CI 는 Dockerfile `bootJar` 만 → **테스트가 CI 에서 안 돌던 공백**을 메움.

- `test-and-report`(ubuntu, JDK 25): `./gradlew test jacocoTestReport` (ArchUnit·추적성·springdoc·REST Docs·Testcontainers 통합 전부) → **JaCoCo·REST Docs snippet·JUnit 결과 artifact 업로드**.
- `karate-flows`(ubuntu, postgres service, JDK 21+25): 앱(Java25 bootRun) 기동 + 테스트 사용자 시드(psql) + `run-flows.sh`(Java21 standalone Karate) → **Karate HTML 리포트 artifact 업로드**.
- 액션 버전은 레포 관례와 일치(checkout@v6.0.2, setup-java@v4, upload-artifact@v7.0.1). 배포판 temurin(=Dockerfile `eclipse-temurin:25`). YAML 문법 검증 통과.

## 검증

- `RequirementsTraceabilityTest` 단독 통과 + 음성 테스트 실패 확인.
- `./gradlew test jacocoTestReport`(JDK25 + Testcontainers) 전체 통과(무회귀).
- 워크플로 YAML `yaml.safe_load` 통과. (GitHub Actions 실행은 로컬 검증 불가 — 첫 PR 실행에서 확인 필요.)

## 메모 / 후속

- `karate-flows` 잡은 앱(25)+Karate(21) 2개 JDK + postgres service + bootRun 백그라운드 구조라, 첫 CI 실행에서 타이밍/환경 조정이 필요할 수 있음. `test-and-report` 가 핵심 테스트 게이트라 그것만으로도 ArchUnit·추적성·springdoc·REST Docs·통합이 CI 에서 강제됨.
- 추적성 확장: 새 요구사항은 requirements.yaml 에 추가 + feature 에 `@REQ` 태그 → 검사기가 자동 강제.
- JaCoCo 커버리지 임계 강제(`jacocoTestCoverageVerification`)는 핵심 패키지 한정으로 추후 도입 가능(전체 % 강제는 지양).
