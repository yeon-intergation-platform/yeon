# 카드 에디터 이미지 등록(업로드/표시) 버그 수정

- 작업자: claude (Opus 4.8)
- 일자: 2026-05-31
- 브랜치: `fix/card-image-upload` (base: `main`)
- 증상: 카드 추가 에디터에서 이미지 등록이 제대로 안 됨

## 진단 — 백엔드에 실제 버그 2개

병렬 코드 매핑(FE 에디터/업로드훅/contract + BE 컨트롤러/서비스/스토리지/설정) 후, 실제 Tomcat을 거치는 Testcontainers 통합 테스트로 두 원인을 실증했다.

1. **(높음) 서블릿 멀티파트 1MB 기본 한도** — `application.yml`/`application-dev.local.yml` 어디에도 `spring.servlet.multipart` 설정이 없어 Spring Boot 기본값(max-file-size **1MB**)이 적용. 프론트는 10MB·서비스는 5MB를 허용한다고 했지만, **서블릿 파서가 1MB 초과 이미지를 컨트롤러 도달 전에 거부** → 대부분의 사진(>1MB) 업로드 실패.
2. **(높음) 이미지 조회 URL의 인코딩된 슬래시(`%2F`)** — `CardDeckAssetService.resolveImageUrl`가 storageKey를 `URLEncoder.encode`해 `imageUrl`이 `/api/v1/card-decks/assets/card-service%2Fimages%2F...` 형태. Spring Security `StrictHttpFirewall`은 `%2F`를 기본 차단(400) → **업로드(201)는 되지만 그 URL로 이미지를 못 불러와 화면에 안 뜸**. Next 프록시(`fetchCardDeckAssetFromSpring`)도 `encodeURIComponent`로 `%2F`를 재생성해 같은 문제.

> 기존 테스트(`CardDeckAssetControllerTests`)는 `@WebMvcTest` + `MockMultipartFile`이라 서블릿 크기 한도도, `%2F` 경로 처리도 검증하지 못해 두 버그가 통과돼 있었다.

부차적 불일치: FE 한도 10MB vs BE 5MB → 5–10MB 이미지가 업로드 후 거부돼 혼란.

## 수정

- **멀티파트 한도**: `application.yml`에 `spring.servlet.multipart.max-file-size: 6MB`, `max-request-size: 10MB` 추가(서비스 5MB 검증이 친절한 에러로 노출되도록 헤드룸).
- **초과 예외 처리**: `MultipartUploadExceptionHandler`(@RestControllerAdvice) 추가 — `MaxUploadSizeExceededException`을 400 + "이미지는 5MB 이하만 업로드할 수 있습니다."로 변환(컨트롤러 도달 전 예외라 전역 advice 필요).
- **`%2F` 제거**: `resolveImageUrl`에서 인코딩 제거 → `/api/v1/card-decks/assets/card-service/images/{hex}.png`(실제 슬래시, `/card-decks/assets/**` 와일드카드가 다중 세그먼트 수용). Next 프록시는 세그먼트 단위 인코딩(`split('/').map(encodeURIComponent).join('/')`)으로 슬래시 보존.
- **한도 정렬**: FE `CARD_EDITOR_MAX_IMAGE_SIZE` 10MB→5MB + 메시지 통일(fail-fast).
- **계약 추가**: `@yeon/api-contract`에 `cardDeckAssetUploadResponseSchema`/타입 추가, `uploadCardDeckImage` 반환을 계약 타입으로 연결.

## 검증 (Playwright + 풀스택)

- 로컬 스택 기동: postgres(docker) + Spring(8081, JDK25, dev.local) + web(3000, 백엔드 연결).
- 풀스택 curl(브라우저→Next→Spring): 2.44MB 업로드 **201**, 반환 imageUrl 되읽기 **200**, 바이트 원본과 동일.
- **Playwright 브라우저 E2E**: 실제 카드 에디터에서 2.44MB PNG 업로드 → 이미지 노드 렌더(`naturalWidth 800`, `complete:true`), 네트워크 **POST 201 + GET 200**. 스크린샷에 노이즈 PNG 표시 확인.

## 잔여 리스크 / 메모

- 환경: 로컬 백엔드 빌드/실행은 Java 25 toolchain 필요. 본 작업에서 SDKMAN `25.0.3-amzn`로 검증(시스템 기본 JDK 미변경, gradle toolchain 자동 감지).
- 과거에 이미 `%2F`로 저장돼 카드 본문에 박힌 imageUrl(있다면)은 레거시로 남음 — 신규 업로드는 정상. 운영 영향 미미(이미지 기능이 사실상 막혀 있었음).
- Karate 실행 러너는 라이브 서버가 필요하고 현 스택(Spring Boot 4/Java 25)에서 의존성 추가 리스크가 있어 build.gradle에 미배선(standalone 실행 문서화). 실증 검증은 Testcontainers 통합 테스트 + Playwright가 담당.
