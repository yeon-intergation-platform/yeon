## 시작 근거

- 운영 SSH 읽기 전용 접속 성공: self-hosted Docker 호스트와 실행 컨테이너 확인.
- `/srv/yeon/.env` 존재 및 권한 `600` 확인. 값은 출력하지 않음.
- 백업: `/srv/yeon/backups/secret-migration/20260710T171840Z`
- 백업 권한: 디렉터리 `700`, 파일 `600`.
- `yeon-2`에는 사용자 변경이 있어 제외하고 깨끗한 `yeon-3`를 사용함.
- 저장소가 public임을 확인하고 `production` Environment를 생성함. custom branch policy는
  `main`만 허용함.
- 운영 값은 `production` Environment Secrets/Variables에 값 노출 없이 동기화함. 이관 성공 뒤
  Repository-level 중복 원천을 제거할 예정임.
- 저장소 기본 workflow 권한을 read로 낮추고 Actions PR 승인 권한을 비활성화함. `main`에는 PR,
  최신 branch, SSOT check, conversation resolve 보호 규칙을 적용함.
- PR/테스트/lint/typecheck/SSOT job을 GitHub-hosted `ubuntu-latest`로 이전함.
- ARM64 image build/publish는 GitHub-hosted `ubuntu-24.04-arm`/`ubuntu-latest`로 분리함. 운영
  `yeon-prod` runner는 deploy workflow만 허용하도록 root 소유 job-start hook을 축소하고 synthetic
  `pull_request`와 build workflow 거부를 확인함.
- 과거 `yeon-ci` runner, rootless Docker와 OS 계정을 제거함.
- Compose v5 environment-source secret 임시 컨테이너에서 restart와 no-recreate 동작을 확인함.
- web/backend/race image ref를 GHCR publish 결과 digest로 고정하는 배포 계약을 구현함.
