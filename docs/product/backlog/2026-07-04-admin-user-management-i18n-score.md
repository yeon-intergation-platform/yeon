# 다국어화 95점 검증과 관리자 회원관리 강화

## 1차

### 작업내용

- 다른 평가 토큰/에이전트로 현재 한국어·영어 다국어화 수준을 점수화한다.
- 95점 미만이면 한글/영문 사용자가 막히는 지점을 찾아 우선 보완한다.
- 기존 `/admin/members`와 Spring `users` 계층을 확장해 운영자가 회원 목록을 실질적으로 관리할 수 있게 한다.
- 회원탈퇴 기능을 사용자가 직접 실행할 수 있게 하고, 관리자도 필요한 경우 회원을 삭제할 수 있게 한다.

### 논의 필요

- 관리자 회원 삭제는 되돌릴 수 없는 파괴적 작업이므로, 실사용 UI에서는 명확한 확인 문구와 자기 자신 삭제 방지가 필요하다.
- 기존 데이터는 FK `ON DELETE CASCADE`가 많은 편이므로, 삭제 전 영향 범위 요약과 세션 무효화가 필요하다.

### 선택지

- A. 기존 `/admin/members`를 확장해 검색, 필터, 권한 변경, 세션 무효화, 삭제를 모두 제공한다.
- B. `/admin/users` 경험치 화면과 `/admin/members`를 하나로 합친다.
- C. 읽기 전용 어드민은 유지하고 별도 위험 작업 페이지를 만든다.

### 추천

- A를 우선한다. 이미 `/admin/members`가 회원 관리 화면이고, 경험치 화면은 상세 보조 화면으로 유지하는 편이 작고 안전하다.

### 사용자 방향

- “유저관리 아주 제대로”, “회원탈퇴기능”, “어드민에서 회원관리하는 잡다한 기능들 전부”라는 방향에 맞춰 A 기준으로 진행한다.

## 2차

### 작업내용

- API 계약에 관리자 회원 상세 필드와 액션 응답을 추가한다.
- Spring에서 관리자 권한 검증, 역할 변경, 세션 무효화, 계정 삭제, 자기 회원탈퇴를 구현한다.
- Next route는 세션/쿠키 브리지와 Spring 호출만 담당한다.

### 논의 필요

- 현재 유지보수 범위는 카드/타자/커뮤니티 중심이므로, 상담 워크스페이스 계열 데이터는 새 기능 확장 대상이 아니다. 단, public.users FK cascade로 인한 삭제는 기존 DB 무결성 정책을 따른다.

### 선택지

- A. hard delete: `public.users` 삭제와 FK cascade에 맡긴다.
- B. soft delete: `deleted_at` 마이그레이션을 추가하고 로그인/목록에서 제외한다.

### 추천

- 현재 스키마에 `deleted_at`이 없고 FK cascade가 이미 광범위하게 정의되어 있어 1차는 A로 구현한다. 단, 삭제 전 세션 무효화와 자기 삭제 방지는 반드시 둔다.

### 사용자 방향

- “회원탈퇴기능”이 즉시 필요하므로 A로 진행한다.
