# 2026-05-17 어드민 회원관리 1차 백로그

## 배경

- deep-interview 결과, 1차 어드민 회원관리는 상담 수강생이 아니라 플랫폼 로그인/인증 유저(`public.users`)를 대상으로 한다.
- 목적은 가입/계정 현황 파악이며, 첫 구현은 read-only 리스트로 제한한다.
- Next.js는 UI/BFF만 맡고, 유저 정보와 권한 판정의 source of truth는 Spring이 담당한다.

## 1차

### 작업내용

- `/admin` 하위에 회원관리 read-only 리스트 화면을 추가한다.
- Spring users 응답에 관리자 리스트에 필요한 `role`, `lastLoginAt`을 포함한다.
- 회원 리스트는 이메일, 표시 이름, role, 가입일, 최근 로그인을 보여준다.
- 관리자만 접근 가능하게 기존 Spring 기반 admin 판정 패턴을 따른다.
- mock/demo 데이터 없이 실제 Spring users API 응답을 사용한다.

### 논의 필요

- 1차 이후 상세 페이지, 권한 변경, 비활성화/삭제, 고급 검색을 어떤 순서로 확장할지.

### 선택지

1. read-only 리스트만 구현하고 모든 write/detail/search 기능은 후순위로 둔다.
2. 리스트와 상세 페이지까지 함께 만든다.
3. 리스트와 role 변경까지 함께 만든다.

### 추천

- 1번. 현재 요구사항은 가입 현황 파악이므로 read-only 리스트만 먼저 만들고, 권한/삭제 같은 위험한 write 기능은 별도 요구사항으로 분리한다.

### 사용자 방향

- 플랫폼 유저 대상.
- 1차 목적은 가입 현황 파악.
- 필수 컬럼은 이메일, 표시 이름, role, 가입일, 최근 로그인.
- 고급 검색, 수정/삭제, 권한 변경, 상세 페이지는 1차 제외.
