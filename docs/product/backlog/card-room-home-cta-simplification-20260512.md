# 카드암기방 홈 문구/CTA 단순화

## 1차수

### 작업내용

- `/card-service` 홈 상단의 영문 라벨 `Card Room` 문구를 제거한다.
- 프로필 섹션의 보조 설명 문단 `타자방과 같은 캐릭터 선택을 사용합니다. 선택한 캐릭터는 카드방 대기/학습 화면에 표시돼요.`를 제거한다.
- 프로필 섹션 제목은 `내 카드방 프로필`이 남아 있는 경로가 있으면 모두 `내 프로필`로 통일한다.
- `오늘의 시작` 영역 CTA는 총 2개만 유지한다.
  - `카드방 입장`
  - 덱 상태 기반 동적 CTA
- `새 덱 만들기` 고정 버튼은 제거한다.
- 덱 상태 기반 동적 CTA 규칙을 적용한다.
  - 덱이 1개 이상 있는 사용자: `내 덱 보기`
  - 덱이 없는 사용자: `새 덱 만들기`
- 동적 CTA의 이동/동작도 문구와 일치시킨다.
  - `내 덱 보기`: `/card-service/decks` 이동
  - `새 덱 만들기`: 기존 `CreateDeckDialog` 열기 또는 새 덱 생성 진입 동작 유지
- 덱 유무 판정은 화면 임의 상태가 아니라 기존 카드덱 목록 source of truth(API/훅/스토어 중 현재 구현에서 쓰는 단일 경로)에서 가져온다.

### 논의 필요

- 홈에서 덱 목록을 즉시 조회할 때 로딩 중 CTA 문구를 어떻게 둘지 결정해야 한다.
- 게스트 로컬 덱과 로그인 서버 덱이 모두 있을 수 있는 경우 `덱 있음` 판정에 둘 다 포함할지 확인이 필요하다.

### 선택지

1. 로딩 중에는 `내 덱 보기`를 기본으로 보여주고, 조회 후 덱이 없으면 `새 덱 만들기`로 바꾼다.
2. 로딩 중에는 CTA를 skeleton/disabled로 두고, 덱 유무 확인 후 `내 덱 보기` 또는 `새 덱 만들기`를 보여준다.
3. 덱 조회 없이 로컬 캐시만 보고 문구를 정한다.

### 추천

- 2번. CTA 문구가 덱 유무의 source of truth를 반영해야 하므로, 로딩 중 임시 문구보다 확인 후 표시가 안전하다.
- 게스트/로그인 혼합 정책을 고려해 기존 카드덱 목록 조회 경로가 게스트 덱을 포함한다면 그대로 사용하고, 포함하지 않는다면 게스트 덱 source까지 함께 보되 판정 함수는 하나로 분리한다.

### 사용자 방향

- 사용자가 요청한 방향 기준으로 진행한다. `오늘의 시작`은 2개 버튼만 남기고, 두 번째 버튼 텍스트는 덱 유무에 따라 동적으로 구성한다.

## 구현 메모

- 예상 변경 파일: `apps/web/src/features/card-service/card-service-home.tsx`
- 기존 CTA 배열 `CARD_HOME_CTA_ITEMS`는 정적 2개 배열 대신 `카드방 입장` 고정 CTA + `deckState` 기반 보조 CTA로 단순화한다.
- `새 덱 만들기` 버튼을 별도 3번째 버튼으로 렌더링하지 않는다.
- UI 변경은 텍스트/CTA 수 정리 범위로 제한하고, 프로필 카드/캐릭터 선택 동작은 유지한다.

## 구현 결과

- `CardServiceHome`에서 기존 덱 목록 source of truth인 `useDeckList`를 사용해 홈 보조 CTA 문구를 결정한다.
- `오늘의 시작`은 `카드방 입장`과 덱 상태 기반 보조 CTA, 총 2개만 렌더링한다.
- 덱 조회 로딩 중에는 `덱 확인 중` disabled CTA를 표시한다.
- 덱이 있거나 덱 조회 오류로 상태를 단정할 수 없으면 `내 덱 보기`를 표시한다.
- 덱이 없으면 `새 덱 만들기`를 표시하고 기존 `CreateDeckDialog`를 연다.
