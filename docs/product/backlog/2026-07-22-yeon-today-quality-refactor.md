사용자의 요청에 따라 서버 기반 YEON Today 구현에서 실제 버그 20개와 구체적인 리팩터링 후보 100개를 찾고, 사용자 상태 정합성과 반복 동작을 해치는 항목부터 수정한다. 수량을 채우기 위한 추상적 제안은 제외하고 각 항목에 코드 위치, 관찰 근거, 영향, 수정 방향을 남긴다.

## 1차 — 기준선과 감사 목록 확정

### 작업내용

- Spring Controller → Service → Repository, Flyway 스키마, 공용 Zod 계약, API client, Next BFF, TanStack Query, 보드·기록 화면과 QA 자동화를 함께 읽는다.
- 재현 가능한 동작 결함 20개를 심각도와 영향 기준으로 분류한다.
- SRP, DRY, CQS, 입력 검증, 실패 경계, 테스트 가능성 기준의 리팩터링 후보 100개를 파일·심볼 단위로 기록한다.

### 논의 필요

- 기존 제품 동작을 바꾸는 기능 확장과 코드 품질 개선을 같은 작업에서 어디까지 허용할지 구분해야 한다.

### 선택지

1. 목록만 문서화하고 구현은 후속으로 미룬다.
2. 목록을 고정한 뒤 P0/P1 버그와 이를 안전하게 고치는 구조 리팩터링까지 같은 작업에서 수행한다.
3. 100개 후보를 모두 즉시 구현한다.

### 추천

2번. 감사 수량은 전부 충족하되, 한 번에 100개 구조 변경을 섞지 않고 사용자 데이터·오류 복구·중복 mutation에 영향을 주는 항목부터 검증 가능한 단위로 고친다.

### 사용자 방향

코드 품질 검사와 개선·리팩터링을 진행하고, 버그 20개와 리팩터링 요소 100개를 찾으라는 요청에 따라 2번으로 진행한다.

## 2차 — 서버·계약·BFF 실패 경계 개선

### 작업내용

- 완료/재개 mutation의 재시도 안전성과 낙관적 동시성 규칙을 바로잡는다.
- API 오류가 HTTP 상태와 서버 오류 코드를 보존하도록 typed error 경계를 만든다.
- Spring 호출 timeout, 환경변수 fallback, BFF path 입력 검증과 민감 응답 로그를 보강한다.
- 동작별 회귀 테스트를 먼저 추가하거나 수정과 동시에 추가한다.

### 논의 필요

- 완료/재개 요청의 동일 상태 재시도를 성공으로 볼지 충돌로 볼지 API 의미를 확정해야 한다.

### 선택지

1. 동일 상태 요청도 stale version이면 항상 `409`로 거절한다.
2. 요청 결과가 이미 달성된 경우 성공으로 응답하고, 반대 상태 변경만 version 충돌로 거절한다.

### 추천

2번. 네트워크 재시도로 같은 명령이 반복돼도 거짓 실패를 만들지 않으며, 실제 경쟁 변경은 계속 `409`로 보호한다.

### 사용자 방향

별도 지시가 없으므로 추천안을 적용한다.

## 3차 — 웹 mutation·오류·접근성 개선

### 작업내용

- 빠른 연속 클릭으로 같은 version 또는 같은 제목 요청이 중복 전송되지 않게 즉시 제출 잠금과 행 단위 pending 상태를 추가한다.
- 성공한 새 작업 전에 과거 mutation 오류를 초기화하고, 쿼리 오류와 인증 오류를 구분한다.
- 완료 시각, 예상 시간 라벨, 캘린더 오류, 비활성 활동 빈 상태를 제품 정의와 일치시킨다.
- 상호작용하지 않는 요소의 잘못된 button/option semantics를 제거한다.

### 논의 필요

- mutation 전체를 잠글지, 동일 엔티티만 잠글지 결정해야 한다.

### 선택지

1. 화면의 모든 mutation을 하나의 전역 pending으로 막는다.
2. 생성 폼은 폼 단위, 할 일·활동은 엔티티 단위로 잠근다.

### 추천

2번. 충돌 가능한 동일 엔티티 동작만 막고 서로 독립적인 작업은 허용한다.

### 사용자 방향

별도 지시가 없으므로 추천안을 적용한다.

## 4차 — 구조 리팩터링과 자동화 안정화

### 작업내용

- 화면 파일에서 순수 표시 모델, 상수, 오류 해석, mutation orchestration을 분리한다.
- `TodayService`와 repository의 조회·명령 책임 분리 후보를 감사 문서에 후속 차수로 남기고, 이번 수정에 필요한 최소 추출만 수행한다.
- QA 스크립트의 고정 sleep, 하드코딩 입력, 브라우저 경로 의존과 실패 수집 범위를 개선한다.

### 논의 필요

- 대형 파일 분해를 이번 버그 수정 PR에서 어느 정도까지 수행할지 결정해야 한다.

### 선택지

1. 버그 수정에 직접 필요한 경계만 추출한다.
2. 감사한 100개 후보를 한 PR에서 전부 분해한다.

### 추천

1번. KISS와 회귀 위험을 고려해 수정에 필요한 추출을 수행하고 나머지는 우선순위가 있는 후속 backlog로 유지한다.

### 사용자 방향

별도 지시가 없으므로 추천안을 적용한다.

## 5차 — 검증·배포

### 작업내용

- Spring Today 테스트, web 단위 테스트, API contract/client 테스트, lint, typecheck, production build를 실행한다.
- Playwright로 생성·중복 방지·완료/재개·날짜 전환·오류/빈 상태와 모바일 overflow를 확인한다.
- 감사 문서의 구현 상태와 남은 후보 우선순위를 갱신한다.
- 변경 파일만 커밋하고 PR(main)을 생성·merge한다.

### 논의 필요

- 없음. 프로젝트 기본 배포 규칙을 따른다.

### 선택지

1. 정적 검사만 통과하면 배포한다.
2. 정적 검사, 관련 테스트, 운영 빌드, 핵심 Playwright 회귀까지 통과한 뒤 배포한다.

### 추천

2번.
