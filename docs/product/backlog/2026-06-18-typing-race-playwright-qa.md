# 타자연습 Playwright 실제 이용 QA

## 배경

로컬 웹과 race-server를 함께 띄운 상태에서 사용자가 실제로 타자방을 만들고, 준비하고, 경기 화면에서 입력하는 흐름을 Playwright로 검증한다. 눈에 보이는 UI/접근성/런타임 오류를 우선 찾고, 즉시 수정 가능한 항목은 같은 차수에서 처리한다.

## 1차: 실제 이용 흐름 재현

### 작업내용

- `/typing-service`, `/typing-service/rooms`, `/typing-service/play`, `/typing-service/practice`를 실제 브라우저로 진입한다.
- 일반 타자방 생성, 방 입장, 시작, 카운트다운 후 입력까지 진행한다.
- 콘솔 오류, 접근성 이름, 모달 역할, 불필요한 사용자 노출 문구를 기록한다.

### 논의 필요

- 없음. 사용자가 직접 Playwright 기반 실사용 시뮬레이션을 요청했다.

### 선택지

- A. 관찰만 하고 별도 이슈로 분리한다.
- B. 재현과 동시에 작은 버그를 바로 고친다.

### 추천

- B. 불필요한 반복 QA를 줄이기 위해 명확한 소규모 버그는 즉시 수정한다.

### 사용자 방향

- 추천 기준으로 진행한다.

## 2차: 우선 수정

### 작업내용

- 실제 레이스 화면에서 개발자용/비활성 기능 문구가 사용자에게 노출되는 문제를 정리한다.
- 방 만들기 모달의 접근성 역할과 버튼 라벨을 확인하고 필요한 속성을 보강한다.
- 정상 사용자가 볼 필요 없는 경고성 API 실패 로그를 조용한 fallback으로 정리한다.
- `/typing-service/practice` hydration 오류가 재현되면 원인을 좁혀 수정한다.

### 논의 필요

- 음성 통화 UI는 기능 활성화 전까지 숨기는 방향이 적절한지.

### 선택지

- A. 비활성 음성 통화 패널을 완전히 숨긴다.
- B. 비활성 상태 안내만 짧게 남긴다.

### 추천

- A. 실제 경기 중 핵심 흐름을 가리는 보조 기능은 활성화 전까지 숨기는 편이 낫다.

### 사용자 방향

- 추천 기준으로 진행한다.

## 3차: 재검증과 배포 준비

### 작업내용

- Playwright로 동일 흐름을 다시 실행해 수정 결과를 확인한다.
- 관련 lint/typecheck 또는 기존 테스트를 실행한다.
- 변경 파일만 커밋하고 main 대상 PR로 정리한다.

### 논의 필요

- 없음.

### 선택지

- A. 로컬 확인까지만 한다.
- B. 프로젝트 정책대로 PR/merge까지 진행한다.

### 추천

- B. 저장소 파일 변경이 있으면 main 배포 흐름까지 완료한다.

### 사용자 방향

- 추천 기준으로 진행한다.
