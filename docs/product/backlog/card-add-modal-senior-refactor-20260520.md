# 카드 추가 모달 시니어 리팩터링 백로그 (2026-05-20)

## 배경

카드 추가 모달은 질문/답변 에디터와 실시간 미리보기 배치 개선 이후 사용성은 좋아졌지만, 직접 작성 폼 내부가 입력 영역과 하단 액션 버튼까지 함께 소유하고 있어 모달 shell과 폼 책임이 섞여 있다. 저장/업로드 중 닫기 요청도 폼 내부 상태를 모달이 알기 어려워 운영 중 반쯤 성공한 사용자 흐름을 막기 어렵다.

## 1차

### 작업내용

- 직접 작성 폼의 저장/취소 액션을 `ResponsiveModal.footer`로 이동한다.
- `AddCardForm`은 입력값, 제출 가능 여부, 저장/업로드 pending, 에러 메시지 계산만 담당한다.
- `AddCardsPanel`은 모달 닫기, dirty 확인, pending 중 닫기 차단, 외부 submit 버튼 연결을 담당한다.
- 기존 데스크톱 우측 실시간 미리보기와 모바일 editor-local 미리보기 동작은 유지한다.
- 신규 백엔드/DB/API 책임은 추가하지 않는다.

### 논의 필요

- 저장/이미지 업로드 중 모달 닫기를 허용할지 차단할지 결정이 필요하다.
- 하단 에러 메시지를 폼 본문에 둘지, 모달 footer에 둘지 결정이 필요하다.

### 선택지

1. 기존처럼 폼 내부 sticky footer 유지
   - 구현이 작지만 모달 shell과 액션 책임 결합이 남는다.
2. 모달 footer로 액션 이동, form id로 외부 submit 연결
   - 구조가 명확해지고 닫기/저장 상태를 모달이 방어할 수 있다.
3. 별도 전역 form controller 훅 도입
   - 확장성은 있지만 현재 범위에 비해 구조가 과하다.

### 추천

2번을 추천한다. 현재 문제는 기능 추가보다 책임 경계와 실패 흐름 방어가 핵심이므로, `ResponsiveModal.footer`를 실제 액션 슬롯으로 사용하고 폼은 상태 계산과 submit handler에 집중시키는 것이 가장 작고 오래 유지되는 수정이다.

### 사용자 방향

추천 기준으로 진행한다. 목표는 “시니어 코드처럼 책임이 분리되고, 저장/업로드 중 닫기 같은 실패 경계가 방어되는 구조”다.
