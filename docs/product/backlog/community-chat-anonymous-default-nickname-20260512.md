# community chat anonymous default nickname 2026-05-12

## 배경

실시간 채팅은 익명성이 핵심이다. 어떤 렌더링 순간에도 로그인 계정의 이름, 이메일 앞부분, 내부 사용자 ID, `senderId` 같은 식별자가 화면에 노출되면 안 된다. 현재 사용자는 “1초라도 유저의 아이디가 나와서는 안 됨”을 명시했으며, 기본 이름은 `익명1231`처럼 익명 + 랜덤숫자 형태여야 한다. 또한 `/community`에서 사용자가 설정한 닉네임은 게시글뿐 아니라 실시간 채팅 전송에도 동일하게 사용되어야 한다.

## 목표

- 실시간 채팅 표시명 source of truth를 계정 정보가 아닌 커뮤니티 익명 닉네임으로 고정한다.
- 첫 방문/초기 렌더에서 기본 닉네임은 `익명` + 랜덤 숫자 형식으로 즉시 준비한다.
- `/community`에서 설정한 닉네임 입력값이 실시간 채팅 전송 payload의 `guestNickname`/`senderNickname`으로 사용되게 한다.
- API 응답이나 레거시 메시지에 계정/ID형 표시명이 섞여도 클라이언트 렌더 직전 방어막으로 익명 표시명만 보여준다.

## 1차: 닉네임 source of truth 정리

### 작업내용

- 커뮤니티 게스트 닉네임 생성/저장 유틸을 만든다.
  - 기본값 형식: `익명${randomNumber}`.
  - 랜덤 숫자는 충분히 넓은 범위로 생성하되 보안 식별자가 아니라 표시명 전용으로만 사용한다.
  - localStorage 또는 기존 커뮤니티 게스트 상태 저장소와 연결해 새로고침 시에도 같은 익명 닉네임을 유지한다.
- `useCommunityFeed`가 쓰는 `guestNickname`을 실시간 채팅에서도 공유 가능한 단일 상태로 끌어올린다.
- `/community` 닉네임 입력값 변경 시 채팅 위젯이 즉시 같은 값을 받도록 `CommunityChatWidget`에 명시적 prop을 추가한다.

### 논의 필요

- 없음. 사용자가 실시간 채팅 익명성 원칙과 닉네임 공유 방향을 확정했다.

### 선택지

1. 기존 게시글 닉네임 상태를 상위 `CommunityPage`에서 채팅 위젯 prop으로 전달한다.
2. 채팅 위젯이 별도 localStorage 키를 읽고 게시글 닉네임과 동기화한다.

### 추천

- 1번. 화면의 닉네임 입력이 source of truth가 되어야 하므로 prop 전달이 가장 단순하고 렌더 타이밍도 안전하다.

### 사용자 방향

- 추천 기준으로 진행한다.

## 2차: 실시간 채팅 전송 payload 익명화

### 작업내용

- `useCommunityChat`의 기본 `DEFAULT_GUEST_NICKNAME = "익명이"`를 제거하고, 상위에서 받은 익명 닉네임을 사용한다.
- 전송 시 `guestNickname`과 `senderNickname`은 항상 동일한 익명 닉네임으로 채운다.
- 닉네임이 비어 있거나 공백이면 즉시 새 익명 랜덤 닉네임을 생성해 사용한다.
- `/api/v1/community-chat/messages` route는 클라이언트가 보낸 `senderNickname`보다 `guestNickname` 또는 서버 기본 익명값을 우선한다.
- route에서는 계정 세션 조회, `displayName`, email, userId를 실시간 채팅 표시명 결정에 절대 사용하지 않는다.

### 논의 필요

- 서버에 이미 저장된 과거 메시지의 `senderNickname`이 계정명일 수 있다. 클라이언트 렌더 방어막에서 숨기되, 별도 데이터 정리는 추후 운영 작업으로 분리할지 결정이 필요할 수 있다.

### 선택지

1. 이번 작업은 신규 전송과 화면 렌더 방어만 처리한다.
2. 과거 저장 메시지 마이그레이션까지 함께 처리한다.

### 추천

- 1번. “1초 노출 금지”는 렌더 방어로 즉시 해결하고, 저장 데이터 정리는 별도 검증이 필요한 운영 작업으로 분리한다.

### 사용자 방향

- 추천 기준으로 진행한다.

## 3차: 1초 노출 방지 렌더 방어막

### 작업내용

- `CommunityChatWidget`에서 `message.senderNickname`을 바로 렌더하지 않는다.
- 렌더 직전 `getAnonymousChatDisplayName(message, fallbackNickname)` 같은 함수로 표시명을 정규화한다.
- 아래 패턴은 계정/ID로 간주해 화면에 표시하지 않고 현재 익명 닉네임 또는 `익명####` fallback으로 대체한다.
  - 이메일 형태 또는 `@` 포함 문자열
  - UUID 형태
  - `user:` / `guest:` / `member:` 같은 내부 ID prefix
  - 과도하게 긴 영문/숫자 토큰
  - 빈 문자열/공백
- `senderId`는 비교용으로도 화면 라벨 결정에 사용하지 않는다.
- `나` 라벨도 제거한다. 본인 메시지 여부는 익명성보다 우선하지 않는다.

### 논의 필요

- 사용자가 직접 설정한 닉네임이 이메일처럼 보이는 경우도 숨겨야 하는가.

### 선택지

1. 실시간 채팅은 강한 익명성 정책이므로 이메일/ID처럼 보이는 닉네임은 입력 단계 또는 렌더 단계에서 익명 fallback으로 대체한다.
2. 사용자가 직접 입력한 닉네임은 그대로 허용한다.

### 추천

- 1번. “1초라도 아이디 노출 금지” 조건에서는 사용자 입력 실수도 방어하는 편이 안전하다.

### 사용자 방향

- 추천 기준으로 진행한다.

## 4차: 테스트/검증

### 작업내용

- route 테스트를 보강한다.
  - 로그인 세션이 있어도 실시간 채팅 payload의 `userId`는 `null`이다.
  - `displayName`, email 앞부분, 클라이언트가 보낸 `senderNickname`이 계정처럼 보여도 최종 `senderNickname`은 익명 닉네임이다.
- 채팅 표시명 정규화 함수 단위 테스트를 추가한다.
  - 이메일, UUID, `user:*`, `guest:*`, 긴 영숫자 ID는 렌더 결과에서 숨겨진다.
  - `익명1234`와 사용자가 설정한 정상 닉네임은 표시된다.
- `/community` 수동 확인 기준을 추가한다.
  - 실시간 채팅 헤더 접속자 수가 초록 점 + 숫자만으로 보인다.
  - 첫 진입 직후 채팅 목록에 계정명/이메일/ID가 보이지 않는다.
  - 닉네임 입력값 변경 후 새 채팅 메시지가 해당 닉네임으로 표시된다.
  - 페이지 새로고침 후에도 기본 익명 랜덤명이 ID로 바뀌지 않는다.

### 논의 필요

- Playwright로 실제 첫 렌더 순간까지 확인할지 여부.

### 선택지

1. 단위/route 테스트 + 수동 확인으로 완료한다.
2. Playwright 스냅샷/DOM 검증을 추가한다.

### 추천

- 1번으로 빠르게 막고, 재발하면 2번을 추가한다.

### 사용자 방향

- 추천 기준으로 진행한다.

## 완료 조건

- 실시간 채팅에서 로그인 계정명, 이메일, 내부 ID가 초기 렌더/폴링/전송 직후 어느 순간에도 표시되지 않는다.
- 기본 채팅 닉네임은 `익명` + 랜덤숫자 형태다.
- `/community` 닉네임 입력값이 실시간 채팅 전송 표시명으로 사용된다.
- 기존 실시간 채팅 전송, 폴링, 접속자 수, 게시글/댓글 기능은 유지된다.
- `/community` 실시간 채팅 헤더의 접속자 표시는 `접속 N명` 텍스트가 아니라 다른 실시간 채팅 UI와 같은 초록 상태 점 + 숫자 형태로 표시한다.
