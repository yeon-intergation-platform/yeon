# 공개 콘텐츠 운영 스냅샷 기반 로컬 검증 및 채널 탐색 개선

## 목표

운영 데이터의 읽기 전용 사본을 격리된 로컬 PostgreSQL과 객체 스토리지에 복원한 뒤, Support·News·Blog 공통 채널 탐색 위치를 실제 콘텐츠 상태에서 검증하고 수정한다.

## 제약

- 이번 작업에서는 PR 생성, push, `main` 병합, 배포를 하지 않는다.
- 운영 DB와 R2에는 읽기 요청만 사용한다.
- 스냅샷과 로컬 환경 파일은 Git 비추적 경로에만 둔다.

## 1차: 운영 사본 로컬 복원

### 작업내용

- 운영 PostgreSQL의 일관된 덤프와 R2 객체 사본을 비추적 로컬 경로에 저장한다.
- 별도 컨테이너와 포트로 로컬 PostgreSQL·객체 스토리지를 구성하고 사본을 복원한다.
- 테이블 수와 객체 수로 복원 결과를 검증한다.

### 논의 필요

- 운영 개인 데이터가 포함된 로컬 사본의 보관 기간과 삭제 정책은 후속 운영 정책으로 결정한다.

### 선택지

1. 운영 원격 환경을 직접 보며 UI를 수정한다.
2. 운영 데이터를 격리한 로컬 사본에서 먼저 검증한다.

### 추천

2번. 운영 쓰기·배포 위험 없이 실제 콘텐츠 밀도와 제목 길이를 기준으로 UI를 검증할 수 있다.

### 사용자 방향

현재 운영의 모든 데이터를 로컬에 적용해 확인하고 수정한다. `main`에는 병합하지 않는다.

### 결과

- 운영 compose의 영속 데이터 원천인 PostgreSQL과 R2 객체 저장소를 읽기 전용으로 사본화해 격리된 로컬 PostgreSQL·MinIO에 복원했다.
- PostgreSQL은 `yeon_backend` 테이블 8개·13MB, 객체 저장소는 45개 객체로 복원 결과를 대조했다.

## 2차: 채널 탐색 위치 검증 및 로컬 수정

### 작업내용

- 공통 헤더는 유지한다.
- Support·News·Blog 전환 요소를 독립된 넓은 띠가 아니라 각 페이지의 첫 콘텐츠 영역에 배치할 수 있는지 로컬 화면에서 비교한다.
- 선택한 구조를 구현하고 Playwright 스크린샷으로 확인한다.

### 논의 필요

- 모바일에서의 압축 표현은 데스크톱 검증 뒤 필요할 때만 추가한다.

### 선택지

1. 공통 헤더 바로 아래에 전폭 탭 띠를 유지한다.
2. 페이지 히어로의 첫 행에 작은 채널 전환을 배치한다.

### 추천

2번. 공통 헤더의 역할을 침범하지 않고, 페이지 목적과 서비스 전환의 관계를 더 자연스럽게 보여 준다.

### 사용자 방향

헤더는 그대로 두고, 현재 탭 위치가 어색한 문제를 로컬에서 확인한 뒤 수정한다.

### 결과

- 공통 헤더는 변경하지 않았다.
- Support·News·Blog 홈은 데스크톱에서 히어로 우측, 모바일에서 소개문 아래에 채널 전환을 배치했다.
- 목록·본문도 각 콘텐츠 시작점에 같은 전환 요소를 배치해 독립된 전폭 탭 띠를 제거했다.
