# 스타 로비 키워드 감지 알림 MVP 백로그

상태: 작업중
작성일: 2026-05-19

## 제품 정의

스타크래프트 유즈맵 방은 피크타임에 몇 초~수십 초 단위로 나타났다가 인원이 차면 바로 사라진다. 이 서비스의 1차 가치는 맵별 커뮤니티나 대기방이 아니라, 사용자가 직접 등록한 방제 키워드가 로비에 뜨는 순간 알려주는 것이다.

```text
유저가 원하는 방제 키워드를 등록해두면,
스타 로비에서 그 키워드가 포함된 방이 관측되는 순간 바로 알려주는 서비스.
```

## 범위 원칙

- 스타 클라이언트 자동 조작, 자동 입장, 게임 내 자동 채팅은 하지 않는다.
- 맵별 대기 채팅/파티/입장 예약은 MVP에서 제외한다.
- 초기 중심 모델은 맵이 아니라 `키워드 알림 조건`이다.
- 맵 별칭/대표 맵 DB는 감지 품질을 높이는 보조 데이터로 뒤에 붙인다.
- 관측 기반 서비스이므로 누락/오인식 가능성을 제품 문구에 명시한다.
- 실제 외부 알림 수단은 브라우저 푸시가 아니라 **Discord 알림**을 우선한다.
- OCR/화면 인식 개발은 실제 스타크래프트 로비 스크린샷 샘플을 사용자에게 받은 뒤에만 진행한다.

## 1차: 키워드 알림 계약과 웹 진입면

### 작업내용

- `/star-lobby` 웹 진입 화면을 만든다.
- 화면은 실시간 로비 피드, 키워드 알림 조건, 관측 신뢰 문구를 먼저 보여준다.
- `packages/api-contract`에 스타 로비 관측 방, 알림 조건, 매칭 이벤트 계약을 추가한다.
- 초기 구현은 route 내부 colocate를 사용하고 `features/star-lobby`는 만들지 않는다.

### 논의 필요

- 비로그인 사용자의 키워드 조건을 로컬 저장으로 허용할지, 로그인 기반 저장만 허용할지 결정이 필요하다.
- Discord 알림을 개인 DM, 사용자 입력 웹훅 URL, 또는 서비스 소유 Discord 봇 중 어떤 방식으로 보낼지 별도 차수에서 결정한다.

### 선택지

1. 브라우저 접속 중 실시간 토스트만 먼저 제공한다.
2. Discord 웹훅/봇 알림을 외부 알림 1순위로 붙인다.
3. 브라우저 푸시 구독을 먼저 붙인다.

### 추천

1차는 `브라우저 접속 중 실시간 토스트 + 서버 저장 알림 조건`으로 시작한다. 외부 알림은 브라우저 푸시가 아니라 Discord를 우선한다.

### 사용자 방향

키워드 감지 기반으로 진행한다. 예: “랜타디 뜨면 바로 알려줘”. 맵별 대기 채팅은 MVP에서 제외한다. 외부 알림은 Discord로 진행한다.

### 진행 기록

- 2026-05-19: `/star-lobby`에 Colyseus `star_lobby` 접속 패널을 추가해 포함/제외 키워드 subscribe, 실시간 관측 피드, 접속 중 `alert_matched` 알림 표시를 연결한다.

## 2차: Spring source of truth

### 작업내용

- `apps/backend/src/main/java/world/yeon/backend/star_lobby/`에 관측 수집, 방 목록 조회, 알림 조건 저장 API를 만든다.
- 관측 데이터와 알림 조건의 영속 저장은 Spring/Flyway가 소유한다.
- 방제 매칭은 포함 키워드, 제외 키워드, 최소/최대 인원 조건으로 시작한다.

### 논의 필요

- 관측 방 보존 기간과 중복 방 판단 기준이 필요하다.
- 방이 사라진 시점을 관측기가 보낼지, 서버가 TTL로 판정할지 결정이 필요하다.

### 선택지

1. 서버 TTL 기반으로 `마지막 관측 후 N초 미갱신`이면 사라짐 처리한다.
2. 관측기가 현재 목록 전체를 보내고 서버가 이전 목록과 diff한다.

### 추천

초기에는 관측기가 현재 목록 전체 snapshot을 보내고 Spring이 diff/TTL을 함께 적용한다.

### 사용자 방향

비어 있으면 추천 기준으로 진행한다.

## 3차: race-server 실시간 이벤트 스트림

### 작업내용

- `apps/race-server/src/star-lobby/`에 로비 이벤트 WebSocket 채널을 추가한다.
- race-server는 실시간 전달 계층만 맡는다.
- Spring에서 확정한 관측/매칭 이벤트를 받아 웹 클라이언트에 push한다.

### 논의 필요

- Spring → race-server 이벤트 전달을 HTTP internal endpoint로 할지, Redis pub/sub로 할지 결정이 필요하다.

### 선택지

1. MVP는 Spring이 race-server internal HTTP endpoint를 호출한다.
2. Redis pub/sub를 먼저 둔다.

### 추천

MVP는 internal HTTP 호출로 시작하고, 트래픽/다중 인스턴스 문제가 생기면 Redis pub/sub로 승격한다.

### 사용자 방향

비어 있으면 추천 기준으로 진행한다.

### 진행 기록

- 2026-05-19: `race-server`에 `star_lobby` Colyseus room과 `POST /internal/star-lobby/events` 내부 publish 엔드포인트를 추가한다. 공개 관측 이벤트는 연결 클라이언트에 브로드캐스트하고, 확정 알림 매칭 이벤트는 `ownerUserId`/`guestSessionId`가 맞는 클라이언트에게만 전달한다.
- 2026-05-19: Spring 관측 수집 후 `STAR_LOBBY_REALTIME_EVENTS_URL`이 설정되어 있으면 race-server 내부 publish 엔드포인트로 `room_observed`, `room_disappeared`, `alert_matched` 이벤트를 전송한다. OCR/자동 클릭 관측기는 스크린샷 샘플 검증 전까지 진행하지 않는다.

## 4차: 관측기

### 작업내용

- `apps/star-lobby-observer/`를 별도 런타임 후보로 둔다.
- 스타 로비 화면에서 방제/인원/관측 시간을 추출해 Spring에 전송한다.
- OCR/이미지 인식 구현은 실제 스타크래프트 로비 스크린샷 샘플을 사용자에게 받은 뒤에만 시작한다.
- 스크린샷 없이 임의 좌표, 임의 클릭 위치, 임의 OCR 영역을 가정해 구현하지 않는다.

### 논의 필요

- 관측기가 돌아갈 OS/환경이 Windows 고정인지 확인이 필요하다.
- 사용자가 제공할 스타 로비 스크린샷 샘플의 해상도, UI 배율, 언어, 방 목록 스크롤 상태가 필요하다.

### 선택지

1. 스크린샷 샘플 수집 후 Python + OpenCV/OCR 후보를 검증한다.
2. 스크린샷 샘플 수집 후 Node/Electron 기반 화면 캡처 후보를 검증한다.
3. 스크린샷 없이 관측기를 구현한다.

### 추천

1번으로 진행한다. 화면 캡처/OCR 튜닝은 실제 로비 이미지가 없으면 신뢰할 수 없으므로, 샘플을 받은 뒤 방 목록 영역·방제·인원 파싱 가능성을 먼저 검증한다.

### 사용자 방향

OCR은 사용자가 스타크래프트 로비 스크린샷을 제공한 뒤 개발한다. 스크린샷 전에는 관측기 구현 대신 수동/더미 관측 입력으로 파이프를 검증한다.

## 5차: 개발용 수동 관측 입력

### 작업내용

- OCR/스타 자동조작 없이 Spring 관측 API에 더미 방 목록을 전송하는 개발용 스크립트를 만든다.
- 기본 샘플 방, `--room`, `--players`, `--file`, `--empty`, `--dry-run` 입력을 지원한다.
- 이 단계의 목적은 `수동 입력 → Spring 저장/매칭 → race-server publish → 웹 알림 표시` 파이프 검증이다.

### 논의 필요

- 실제 운영 관측기의 OS/화면 샘플은 아직 확정하지 않는다.

### 선택지

1. CLI 스크립트로 먼저 만든다.
2. 관리자 웹 UI를 먼저 만든다.

### 추천

초기에는 CLI 스크립트가 안전하다. 웹 관리자 UI는 인증/권한/노출 경계가 추가되므로 뒤로 미룬다.

### 사용자 방향

OCR/자동조작 전 단계로, 수동/더미 관측 입력부터 진행한다.

## 6차: 서버 저장 알림 조건 UI

### 작업내용

- `/star-lobby` 실시간 알림 폼에서 입력한 포함/제외 키워드를 Spring 알림 조건 API에 저장한다.
- Next.js는 `/api/v1/star-lobby/alert-rules` BFF 브리지만 제공하고, 알림 조건의 원천과 검증은 Spring에 둔다.
- 비로그인 사용자는 브라우저 게스트 세션 ID를 헤더로 전달해 같은 브라우저에서 저장 조건을 다시 불러온다.
- 저장과 동시에 race-server 실시간 구독도 갱신해 접속 중 감지는 즉시 동작하게 한다.

### 논의 필요

- 저장된 알림 조건 삭제/수정 UI는 이번 차수에 포함하지 않는다.
- 브라우저 푸시 권한/구독 관리는 별도 차수에서 결정한다.

### 선택지

1. 저장된 조건 목록 조회와 신규 저장만 먼저 연결한다.
2. 수정/삭제/푸시 권한까지 한 번에 붙인다.

### 추천

1번으로 진행한다. 지금 필요한 검증은 “유저가 키워드를 저장하고, 같은 세션에서 조건을 재사용할 수 있는가”이다.

### 사용자 방향

비어 있으면 추천 기준으로 진행한다.

## 다음 개발 고정 계획

현재 MVP 파이프는 `수동 입력 → Spring 저장/매칭 → race-server 실시간 전달 → 웹 표시`까지 연결되어 있다. 다음 작업은 아래 순서로 진행한다.

### 10차: 현재 열린 방 목록 화면

#### 작업내용

- `/star-lobby` 첫 진입 시 Spring `GET /api/v1/star-lobby/rooms`에서 현재 관측된 방 목록을 불러온다.
- 마지막 관측 시각, 관측 기반 안내 문구, 사라진 방/오래된 관측 상태를 표시한다.
- 수동 관측 스크립트로 넣은 방이 새로고침 후에도 현재 방 목록에 보이는지 확인한다.

#### 논의 필요

- 방 목록에서 노출할 최대 개수와 오래된 관측 기준 초 단위는 실제 관측 주기 확정 전까지 임시값으로 둔다.

#### 선택지

1. 현재 방 목록/마지막 관측 시각만 먼저 붙인다.
2. 맵별 집계/시간대 통계까지 같이 붙인다.

#### 추천

1번으로 진행한다. OCR 전에도 수동 관측 입력으로 제품의 기본 화면을 검증할 수 있다.

#### 사용자 방향

비어 있으면 추천 기준으로 진행한다.

### 11차: 알림 조건 관리

#### 작업내용

- 저장된 알림 조건 삭제, 끄기/켜기, 중복 조건 방지, 최소/최대 인원 조건 UI를 추가한다.
- 조건 원천은 Spring에 두고 Next는 BFF 브리지만 제공한다.

#### 논의 필요

- 수정 UI를 인라인으로 할지, 삭제/끄기만 먼저 제공할지 결정이 필요하다.

#### 선택지

1. 삭제와 enabled 토글만 먼저 제공한다.
2. 이름/키워드/인원 조건 수정까지 한 번에 제공한다.

#### 추천

1번으로 진행한다. 저장 조건이 쌓였을 때 사용자가 정리할 수 있는 최소 관리 기능이 먼저 필요하다.

#### 사용자 방향

비어 있으면 추천 기준으로 진행한다.

### 12차: Discord 알림

#### 작업내용

- 외부 알림 수단은 Discord로 고정한다.
- 키워드 매칭이 발생하면 Discord로 “방 떴다” 알림을 보낸다.
- 브라우저 푸시는 MVP 외부 알림 1순위가 아니다.

#### 논의 필요

- Discord 알림 전달 방식을 결정해야 한다: 사용자별 웹훅 URL 저장, 서비스 Discord 봇 DM, 특정 서버/채널 구독 중 하나.
- Discord 계정/웹훅 정보 저장 시 암호화와 삭제 플로우가 필요하다.

#### 선택지

1. 사용자 입력 Discord 웹훅 URL로 먼저 보낸다.
2. Discord 봇 OAuth/DM 구조를 먼저 만든다.
3. 서비스 공용 Discord 서버 채널 구독으로 시작한다.

#### 추천

초기 검증은 1번이 가장 작다. 다만 웹훅 URL은 민감 정보이므로 Spring이 저장 원천과 암호화/삭제를 담당해야 한다.

#### 사용자 방향

Discord로 진행한다. 브라우저 푸시를 우선하지 않는다.

### 13차: 스크린샷 기반 OCR 관측기 PoC

#### 작업내용

- 사용자가 제공한 실제 스타크래프트 로비 스크린샷을 기준으로 방 목록 영역, 방제, 인원 위치를 식별한다.
- 최소 5~10장 이상의 다른 상황 샘플로 OCR 정확도를 확인한다.
- 성공 기준을 만족할 때만 관측기 런타임 구현으로 넘어간다.

#### 논의 필요

- 필요한 스크린샷: 빈 로비, 방 여러 개, 인원 숫자가 다른 방, 한글/영문 방제, 스크롤이 필요한 목록, 피크타임 목록.
- 자동 로그인/자동 클릭은 MVP 범위가 아니며, 관측기는 로비 화면을 읽는 역할로 제한한다.

#### 선택지

1. 스크린샷 샘플 기반 OCR 파싱 PoC를 먼저 한다.
2. 바로 실행형 관측기 앱을 만든다.

#### 추천

1번으로 진행한다. 스크린샷 없이 좌표와 OCR 영역을 가정하면 실제 스타 로비에서 동작할 가능성을 검증할 수 없다.

#### 사용자 방향

OCR은 사용자가 스타크래프트 로비 스크린샷을 제공한 뒤 개발한다.

## 7차: 운영 backend 부팅 실패 복구

### 작업내용

- 운영 배포에서 `StarLobbyRealtimePublisher` 빈 생성이 기본 생성자 탐색으로 실패하는 문제를 복구한다.
- 운영 생성자를 명시적 Spring 주입 생성자로 고정해 테스트용 생성자와 충돌하지 않게 한다.
- backend 단위 검증으로 Spring 컨텍스트 부팅 경계를 확인한다.
- Docker 배포 워크플로에서 backend 변경 시 운영 컨테이너 교체 전에 새 이미지의 1회성 actuator health preflight를 통과하도록 한다.

### 논의 필요

- 없음. 운영 backend가 부팅하지 못하는 배포 차단 버그이므로 즉시 복구한다.

### 선택지

1. 운영 생성자에 `@Autowired`를 명시한다.
2. 테스트용 생성자를 별도 factory/helper로 분리한다.
3. 기본 생성자를 추가하고 필드를 후주입한다.

### 추천

1번으로 진행한다. 현재 실패 원인은 생성자 후보가 명확하지 않은 점이며, 필수 의존성은 생성자 주입으로 유지해야 한다.

### 사용자 방향

운영 오류를 해결하고 main에 머지한다.

## 8차: web/race-server 배포 전 교체 안전장치 확장

### 작업내용

- backend에만 적용된 운영 교체 전 preflight를 web과 race-server에도 확장한다.
- 새 web 이미지는 실제 운영 컨테이너 교체 전에 1회성 컨테이너로 부팅하고 `/api/health` 응답을 확인한다.
- 새 race-server 이미지는 실제 운영 컨테이너 교체 전에 1회성 컨테이너로 부팅하고 `/health` 응답을 확인한다.
- preflight 실패 시 기존 운영 컨테이너를 유지하고 deploy job만 실패시킨다.

### 논의 필요

- 없음. 운영 안정성 요구사항이므로 backend와 같은 배포 차단 기준을 web/race-server에도 맞춘다.

### 선택지

1. 배포 워크플로에서 web/race-server 1회성 컨테이너 preflight를 추가한다.
2. compose healthcheck만 추가한다.
3. blue/green 또는 canary 배포 구조로 전환한다.

### 추천

1번으로 진행한다. 현재 배포 구조를 크게 바꾸지 않으면서 운영 컨테이너 교체 전 새 이미지의 런타임 부팅 실패를 막을 수 있다.

### 사용자 방향

web/race-server도 backend와 마찬가지로 자동 배포 안전장치를 적용하고 main에 머지한다.

## 9차: backend preflight entrypoint 오용 복구

### 작업내용

- backend preflight가 `--entrypoint /bin/sh`로 원래 Java entrypoint를 우회해 `/bin/sh: java: not found`로 실패하는 문제를 복구한다.
- backend도 web/race-server와 같은 1회성 컨테이너 실행 후 container IP의 `/actuator/health`를 확인하는 공통 preflight 경로를 사용한다.

### 논의 필요

- 없음. 새 이미지 교체 전 검증 자체는 맞지만 실행 방식이 잘못되어 즉시 복구한다.

### 선택지

1. backend 전용 shell preflight를 제거하고 공통 HTTP preflight를 사용한다.
2. shell 안에서 Java 절대 경로를 찾아 직접 실행한다.
3. 별도 preflight용 entrypoint 스크립트를 이미지에 추가한다.

### 추천

1번으로 진행한다. 이미지의 원래 `ENTRYPOINT ["java", "-jar", "/app/app.jar"]`를 그대로 검증해야 운영 실행 방식과 같은 조건을 확인할 수 있다.

### 사용자 방향

backend preflight 실패를 복구하고 main에 머지한다.
