# 커뮤니티 화면 품질 개선 사이클

## 배경

커뮤니티 서비스는 게시글 목록, 작성/수정/삭제, 댓글, 실시간 채팅, 게스트 신원 확인이 한 화면에 모여 있다. 실제 브라우저에서 핵심 흐름을 확인하고, P0/P1 문제 중 가장 작은 독립 묶음을 수정한다.

## 1차

### 작업내용

- 범위를 커뮤니티 서비스로 고정한다.
- Playwright로 첫 진입, 비로그인/게스트 상태, 작성, 수정, 삭제, 새로고침, 모바일 viewport, 실패 입력, 빈/에러/로딩 상태를 가능한 범위에서 확인한다.
- 핵심 화면 스크린샷을 `ai-log/hyeonjun/2026-06-19/community-visual-quality-cycle-screenshots/`에 남긴다.
- P0/P1/P2를 분류하고, P0/P1 중 가장 작은 독립 묶음만 수정한다.
- 재발 방지 장치를 최소 1개 추가한다.

### 논의 필요

- 인증/권한/DB 마이그레이션이 필요한 문제는 이번 UI 품질 사이클 범위를 넘긴다.

### 선택지

- 선택지 A: 커뮤니티 전체를 재설계한다.
- 선택지 B: 실제 사용 흐름에서 확인된 P0/P1 중 작은 묶음만 개선하고, 디자인 시스템은 유지한다.
- 선택지 C: 스크린샷 평가만 남기고 코드 수정은 하지 않는다.

### 추천

- 선택지 B. 절차의 범위를 지키면서도 사용자가 보는 품질을 실제로 개선할 수 있다.

### 사용자 방향

- 사용자가 "고쳐줘"와 함께 20단계 서비스 품질 루프를 지시했으므로, 선택지 B로 진행한다.

### 결과

- P1: 커뮤니티 feed BFF가 단독 web dev 실행에서 Spring 내부 토큰을 보내지 못해 목록/게스트 작성 경로가 403으로 막히던 문제를 수정한다.
- P1: feed 오류 상태와 empty 상태가 동시에 노출되어 원인 판단이 흐려지는 문제를 수정한다.
- 요청 반영: 홈 서비스 카드의 `운영 중` 배지를 더 명확한 emerald 계열로 조정한다.
- 재발 방지: Spring BFF 공통 헤더 빌더의 로컬 기본 토큰 fallback, chat-service auth/feed 클라이언트 헤더 테스트, Karate seed 전 `public.users` preflight를 추가한다.
