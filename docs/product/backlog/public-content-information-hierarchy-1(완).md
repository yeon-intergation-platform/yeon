# 공개 콘텐츠 정보 위계 개선 1차 백로그

> 상태: 완료 (2026-07-14)

## 1차 — 서비스 목적과 탐색 흐름 분리

### 작업내용

- 공통 헤더는 변경하지 않고, Support·News·Blog 본문 첫 화면에서 각 서비스의 목적을 더 직접적인 문구로 구분한다.
- Support의 첫 화면을 검색 → 서비스 선택 → 자주 찾는 문제 → 서비스별 문서 → 문의 순서로 재배치한다.
- Support 검색은 제목·설명·요약·본문을 실제로 검색하고 URL의 `q` 상태를 유지한다.

### 논의 필요

- 자연어 의도 분류, 오타 교정, 자동완성은 현재 정적 공개 문서 구조만으로 신뢰성 있게 제공하기 어렵다.

### 선택지

1. 검색창을 시각 요소로만 추가한다.
2. 현재 공개 문서 데이터 범위에서 동작하는 서버 렌더링 검색을 추가한다.

### 추천

- 2번. 보이는 입력창과 실제 결과가 불일치하지 않으며 별도 백엔드 소유권도 만들지 않는다.

### 사용자 방향

- 헤더는 그대로 둔다. 세 사이트의 목적이 외형까지 같아 보이지 않게 본문 정보 위계만 개선한다.

## 2차 — 카드와 목록 위계 정리

### 작업내용

- 공통 문서 카드의 서비스·분류, 제목, 설명, 날짜·읽기 시간을 서로 다른 위계로 배치한다.
- Support 서비스별 문서는 대표 4개만 노출하고 전체 보기 링크를 제공한다.
- News는 중요 공지 하나와 필터 가능한 최신 소식 목록으로 단순화한다.
- Blog는 과도한 상단 여백을 줄이고, 글 성격별 이동을 압축된 탐색으로 제공한다.

### 논의 필요

- 문서 제목 전면 개편이나 영향도/조치 필요 여부의 데이터 모델 추가는 실제 콘텐츠 검수 범위를 크게 넓힌다.

### 선택지

1. 모든 문서 제목과 메타데이터 원본을 함께 다시 쓴다.
2. 원본 콘텐츠는 보존하고, 카드 위계와 이미 검증된 서비스·분류 정보를 중심으로 개선한다.

### 추천

- 2번. 검증되지 않은 공지 영향도나 문서 의미를 새로 추정하지 않고도 탐색 효율을 높인다.

### 사용자 방향

- 현재의 무채색, 가는 테두리, 넓은 여백, 편집물 같은 분위기는 유지한다.

## 3차 — 브라우저 검증과 main 반영

### 작업내용

- Support·News·Blog desktop(1440px)과 mobile(375px) 전후 화면을 Playwright로 비교한다.
- 검색 URL 상태, 서비스·카테고리 이동 링크, 키보드 포커스를 확인한다.
- 관련 단위 테스트, lint, typecheck, build를 통과시키고 PR(main)을 생성·병합한다.

### 논의 필요

- 필수 GitHub 검사가 진행 중이면 보호 규칙을 우회하지 않고 자동 병합 또는 통과 후 병합한다.

### 선택지

1. 정적 화면만 확인한다.
2. 실제 검색·링크 동작과 반응형 화면을 함께 확인한다.

### 추천

- 2번. 이번 변경의 핵심은 시각 변화뿐 아니라 탐색 흐름이기 때문이다.

### 사용자 방향

- 추천 기준으로 구현하고, 변경 후 main에 반영한다.

## 결과

- 공통 헤더는 변경하지 않고 Support·News·Blog 본문 정보 위계만 분리했다.
- Support에는 URL `q` 상태를 유지하는 실제 문서 검색과 문제 해결 순서를 적용했다.
- News는 중요 공지와 단일 최신 소식 목록으로, Blog는 압축된 글 분류 탐색으로 정리했다.
- 자동완성·오타 교정·추정 기반 영향도 라벨·문서 제목 전면 개편은 근거 부족과 과도한 범위를 이유로 적용하지 않았다.
