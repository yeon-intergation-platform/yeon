- 논의 필요: 모든 클릭을 다 추적하기보다 운영 판단에 바로 쓰는 퍼널·진입·생성·학습 시작 이벤트를 우선 표준화하고, 세부 UX 이벤트는 후속 차수로 나눌지 결정한다.

- 추천: 옵션 A. 현재는 데이터 가치가 높은 진입·로그인·생성·시작 이벤트를 먼저 안정적으로 심고, 노이즈성 이벤트는 운영 데이터 확인 후 확장하는 편이 안전하다.

### 차수 8 — 화면 노출형 SEO 카피 롤백 + 메타데이터 기반 SEO 유지

- 작업내용: `typing-service`, `card-service`, 루트 서비스 허브 등 실제 화면에 노출된 SEO 목적성 카피와 FAQ/보조 설명을 제품 중심 문구로 되돌리고, 검색 노출은 `metadata`, `keywords`, `FAQ JSON-LD`, `robots`, `sitemap` 등 비가시 SEO 자산으로 유지한다.
- 논의 필요: FAQ를 화면에 다시 둘 때는 SEO 목적이 아니라 실제 도움말 섹션으로 재설계할지, 계속 구조화 데이터 전용으로만 유지할지 결정이 필요하다.
- 선택지:
  - 옵션 A. 화면에서는 SEO형 설명을 제거하고, 검색 신호는 메타데이터/구조화 데이터에 집중한다.
  - 옵션 B. 화면에도 도움말/설명 섹션을 유지하되 제품 문구로 전면 재작성한다.

- 추천: 옵션 A. 사용자가 화면 노출 없는 SEO 개선을 원했고 현재 목표와 정확히 맞는다.

### 차수 12 — Search Console 등록 전 SEO 표면 정합성 점검

- 작업내용: `yeon.world`, `typing.yeon.world`, `card.yeon.world`, `community.yeon.world` 공개 SEO 표면의 redirect, canonical, robots, sitemap 정책을 실제 운영 응답과 맞춘다.
- 논의 필요: Search Console은 Domain property 하나로 통합 진단을 볼지, URL-prefix property를 서비스별로 나눠 사이트맵 처리 상태를 더 세밀하게 볼지 운영 기준을 정해야 한다.
- 선택지:
  - 옵션 A. Domain property와 서비스별 URL-prefix property를 함께 등록하고, 각 host의 sitemap은 자기 host canonical URL만 담게 한다.
  - 옵션 B. `yeon.world/sitemap.xml` 하나에 모든 subdomain URL을 cross-submit한다.
- 추천: 옵션 A. Google은 verified site 간 cross-site sitemap을 지원하지만, 서비스별 URL-prefix와 host별 sitemap을 맞추면 제출/오류 추적이 더 명확하다.
- 사용자 방향: 추천 기준으로 진행.
