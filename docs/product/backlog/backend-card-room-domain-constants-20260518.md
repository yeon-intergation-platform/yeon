# 백엔드 카드방 도메인 상수화 1차

## 배경

카드방 백엔드 서비스가 JPA Entity를 쓰지 않는 방향은 유지하되, 상태/역할/결과/메시지 타입/ID prefix 같은 도메인 값이 문자열로 서비스와 repository에 흩어져 있다. 이 구조는 상태 전이와 권한 검증에서 오타를 만들기 쉽고, 리뷰 시 source of truth를 바로 찾기 어렵다.

## 1차

### 작업내용

- pre-commit 전체 typecheck가 모바일 Node 타입 로딩 누락으로 실패하지 않게 모바일 tsconfig의 Node 타입 entrypoint도 함께 보정한다.

- 카드방 도메인 값 객체/enum을 `card_rooms.domain` 아래에 둔다.
- `CardRoomService`의 상태 전이, 역할 판단, 결과 검증, 메시지 타입, public id prefix를 도메인 타입으로 치환한다.
- `CardRoomRepository`의 삽입/수정 파라미터도 가능한 곳은 도메인 타입을 받아 DB 값으로만 변환한다.
- 기존 DTO/DB 문자열 계약은 유지해 외부 API 호환성은 깨지지 않게 한다.

### 논의 필요

- 카드방 외 다른 백엔드 도메인까지 한 PR에 모두 정리할지 여부.

### 선택지

1. 카드방 도메인부터 작게 정리하고 이후 반복 패턴을 다른 도메인에 적용한다.
2. 백엔드 전체를 한 번에 대규모 상수화한다.

### 추천

- 1번. 지금 카드방은 상태 전이/자동삭제/실시간 연결과 직접 연결되어 있어 오타 방지가 실제 제품 안정성에 가장 크다. 전체 백엔드 동시 정리는 충돌과 회귀 위험이 크다.

### 사용자 방향

- 추천 기준으로 진행한다.
