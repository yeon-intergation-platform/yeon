# 카드 덱 기반 백지 학습 2차 구현 백로그

작성일: 2026-07-11
상태: 구현·로컬 검증 완료, 운영 배포 대기
서비스: `card-service` 연동 백지 학습 (`blurt.yeon.world`)

## 제품 정의

백지는 카드 서비스의 덱을 학습 재료로 사용한다. 사용자는 먼저 덱을 선택하고
카드의 질문(front)을 본 뒤 답(back)을 기억으로 전부 쓴다. 제출 후 서버가 의미를
채점하고 시도 이력과 SRS 일정을 함께 저장한다.

- 질문 없는 빈 화면 회상과 타자 지문 첫 글자 힌트는 사용하지 않는다.
- 카드 덱과 카드는 하나의 원본만 두며 백지가 복제 소유하지 않는다.
- 덱 선택 화면에서 질문과 답을 바로 확인할 수 있어야 한다.
- AI는 저장 전 편집 가능한 덱 초안을 만들고, 사용자가 확인한 뒤 한 번에 저장한다.
- 비로그인 사용자는 로컬 게스트 덱으로 핵심 학습을 체험하되 유료 외부 AI 호출과
  서버 이력 저장은 로그인 사용자에게만 제공한다.

## 소유권과 경계

- 장기 상태, 권한 판정, AI 생성·채점, 사용량 제한, 시도 이력, SRS 갱신은 Spring이
  소유한다.
- `packages/api-contract`가 DTO와 상태값의 원본이며 웹과 모바일은 이를 사용한다.
- 웹 Route Handler가 필요하면 인증 쿠키를 Spring에 전달하는 얇은 BFF만 맡는다.
- 게스트 데이터는 기존 카드 서비스 로컬 저장소 포트를 재사용하며 서버 상태처럼
  가장하지 않는다. 브라우저 same-origin 정책상 `blurt.yeon.world`와
  `card.yeon.world`의 비로그인 저장소는 직접 공유하지 않으며, 로그인 뒤 명시적
  계정 이관만 두 서비스·기기 간의 연결 수단으로 제공한다.
- 외부 AI 채점은 DB 트랜잭션 밖에서 끝내고, 확정된 채점 결과·시도 이력·SRS 변경을
  하나의 Spring 트랜잭션으로 저장해 장시간 DB 잠금을 피한다.

## 1차 - 계약, 인증, 사용량 제한

- **작업내용**: 백지 세션·채점·이력·AI 덱 초안 계약을 `api-contract`에 추가하고
  `api-client`에 공용 클라이언트를 만든다. 모든 AI 엔드포인트는 로그인 사용자만
  허용하고 사용자별 요청 제한과 명확한 `429` 오류 계약을 둔다. 기존 공개 Next
  채점 Route Handler는 Spring 인증 BFF로 축소한다.
- **논의 필요**: 게스트에게 서버 AI 채점을 허용할지 여부.
- **선택지**: (a) 게스트도 익명 AI 허용 (b) 로그인 사용자만 AI 허용하고 게스트는
  로컬 직접 대조·자기 평가 사용.
- **추천**: (b). 비용 남용과 익명 우회 공격을 막고 로그인 전에도 핵심 UX는 체험할
  수 있다.
- **사용자 방향**: 전체 구현 진행. 추천안 기준으로 구현한다.

## 2차 - 덱 선택과 질문·답 미리보기

- **작업내용**: 백지 홈에서 학습 덱을 먼저 선택하게 하고, 선택과 동시에 카드의
  질문·답을 인라인으로 확인한다. 카드 서비스 덱 상세로 이동할 수 있으며 질문이
  없는 카드는 세션 대상에서 제외하고 이유를 표시한다.
- **논의 필요**: 미리보기에서 답을 기본 공개할지 여부.
- **선택지**: (a) 질문·답 모두 공개 (b) 답은 펼치기 전까지 숨김.
- **추천**: (a). 이 화면은 시험이 아니라 학습 범위 확인 단계이며 사용자가 덱을
  검수할 수 있어야 한다.
- **사용자 방향**: 질문·답을 동시에 확인할 수 있게 구현한다.

## 3차 - Spring AI 덱 생성과 편집 후 저장

- **작업내용**: 주제 또는 원문을 입력하면 Spring이 구조화된 질문·답 카드 초안을
  생성한다. 응답은 저장하지 않고 클라이언트 편집 화면에 반환한다. 사용자는 덱
  이름, 설명, 질문, 답을 수정·추가·삭제한 뒤 기존 카드 덱 bulk 쓰기 경계로 한 번에
  저장한다. 생성·저장 권한과 입력 길이·카드 수를 검증한다.
- **논의 필요**: AI가 곧바로 DB에 저장할지 여부.
- **선택지**: (a) 즉시 저장 (b) 초안 미리보기·편집 후 명시적으로 저장.
- **추천**: (b). 잘못된 생성물이 원본 덱을 오염시키지 않는다.
- **사용자 방향**: 카드 서비스처럼 AI로 한 번에 만들되 저장 전 확인·편집한다.

## 4차 - 의미 채점, 시도 이력, SRS 원자성

- **작업내용**: 질문을 항상 표시하고 사용자가 답을 전부 작성해 제출한다. Spring이
  기준 답과 사용자 답을 의미 채점한 뒤 확정 결과, 시도 이력, 카드 SRS 일정을 하나의
  트랜잭션으로 저장한다. 중복 제출 키를 지원해 재시도로 이력이나 SRS가 두 번
  갱신되지 않게 한다. 채점 실패 시 상태를 쓰지 않는다.
- **논의 필요**: 외부 AI 호출을 DB 트랜잭션 안에 둘지 여부.
- **선택지**: (a) 호출까지 트랜잭션에 포함 (b) AI 결과를 먼저 확정하고 DB 쓰기만
  단일 트랜잭션으로 수행.
- **추천**: (b). 원자성을 유지하면서 네트워크 지연으로 DB 연결과 잠금을 오래
  점유하지 않는다.
- **사용자 방향**: 추천안 기준으로 채점 결과·이력·SRS 저장을 원자적으로 구현한다.

## 5차 - 게스트와 모바일 패리티

- **작업내용**: 웹 게스트 카드 저장소의 덱을 백지에서 선택·미리보기·학습하게 한다.
  게스트 결과와 학습 진행은 로컬 저장소에 기록한다. 공용 계약, query key, 세션
  view-state, 저장소 포트를 패리티 레지스트리에 등록하고 모바일에도 덱 선택,
  질문·답 미리보기, 질문 기반 답 작성, 결과 흐름을 제공한다. 플랫폼별 경로와 저장
  백엔드는 명시적으로 다르게 둔다.
- **논의 필요**: 모바일 게스트 데이터를 웹과 자동 동기화할지 여부.
- **선택지**: (a) 기기 로컬만 (b) 계정 없는 기기 간 동기화 추가.
- **추천**: (a). 로그인 없는 동기화는 별도 식별·보안 설계가 필요하므로 범위를
  넓히지 않는다.
- **사용자 방향**: 웹 게스트와 모바일 핵심 기능 패리티까지 구현한다.

## 6차 - 검증, 배포 안정성, 운영 확인

- **작업내용**: 계약·Spring 서비스·저장소·인증·제한·중복 제출·게스트·모바일
  테스트를 추가한다. lint, typecheck, backend test, parity 검증, production build를
  통과시킨다. 웹은 Playwright로 덱 선택부터 결과까지 확인하고 스크린샷 근거를 남긴다.
  GHCR 인증 실패는 운영 시크릿 전환 작업에서 이미 추가한 backoff 재시도 하네스로
  검증한다. PR을 main에 머지하고 운영 E2E로 종료한다.
- **논의 필요**: AI 공급자 장애 시 자동 축자 채점으로 통과 처리할지 여부.
- **선택지**: (a) 자동 축자 fallback (b) 저장하지 않고 재시도 가능한 실패로 표시.
- **추천**: (b). 의미 답변을 축자 기준으로 잘못 판정해 학습 상태를 오염시키지 않는다.
- **사용자 방향**: 전체 검증·머지·운영 확인까지 진행한다.

## 완료 조건

- 공개 AI 채점 엔드포인트가 없고 인증과 사용자별 제한이 테스트로 증명된다.
- 로그인·게스트 모두 질문이 있는 카드 덱을 먼저 선택하고 내용을 확인할 수 있다.
- AI 덱 초안을 편집한 뒤 카드 서비스의 실제 덱으로 저장할 수 있다.
- 채점 성공 한 번당 시도 이력과 SRS가 정확히 한 번 함께 갱신된다.
- 웹과 모바일이 공용 계약을 사용하고 패리티 검사가 통과한다.
- 전체 CI, Playwright, 운영 E2E가 통과하고 관련 PR이 main에 머지된다.

- 1~5차 계약·Spring·web·mobile 구현을 완료했다.

- web 1,100 tests, mobile 전체 28 tests, 계약 71 tests와 Universal UI parity가
  통과했다.

  정답 대조·자기 평가 → 완료 요약을 desktop/mobile viewport에서 확인했다.
