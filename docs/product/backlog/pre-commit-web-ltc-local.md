# 커밋 단계에서 lint/typecheck/build 강제 실행 및 PR 게이트 제외 백로그 (2026-05-13)

## 배경

- 최근 작업은 `pnpm --filter @yeon/web lint`, `pnpm --filter @yeon/web typecheck`, `pnpm --filter @yeon/web build`를 CI에서 별도 게이트로 돌리기보다
  사용자 로컬 커밋 시점에 선처리하여 병목을 줄이려는 요청이 있었다.

## 차수 1

### 작업내용

1. 커밋 훅(`.githooks/pre-commit`)에 `pnpm --filter @yeon/web build` 실행을 추가해 `lint/typecheck/build`를 한 번에 커밋 전 선검증.
2. 기존 SSOT/스킬 wrapper 점검은 유지하고, 실패 시 커밋 블록 로직은 유지.

### 논의 필요

- 커밋 시 항상 `build`를 실행할지, 웹 관련 변경분이 있을 때만 실행할지.

### 선택지

- A. 변경 경로와 무관하게 항상 3개 실행
- B. `apps/web` 또는 `.next` 영향 파일 변경 시에만 3개 실행

### 추천

- 초기에는 A로 적용해 동작 일관성을 확보하고, 필요 시 B로 조정.

### 사용자 방향

- 기본안(A) 선호

## 차수 2

### 작업내용

1. 동일 의도 문서화 유지(백로그/로그).
2. 커밋 검증(SSOT/sync-skills) + 훅 동작 확인.

### 논의 필요

- 없음.

### 선택지

- 기존 SSOT 점검을 PR에서 유지
- PR 게이트는 최소화

### 추천

- PR 게이트 최소화(필요 시 push 시점 검증).

### 사용자 방향

- PR 게이트 최소화

## 차수 3

### 작업내용

1. `ci`와 로컬 커밋 훅에서 `pnpm --filter @yeon/web build`를 제거.
2. `build`는 배포 파이프라인의 이미지 빌드 단계에서 담당되도록 유지해, 로컬 커밋 생산성을 회복.
3. 사용자 입력을 반영해 훅은 `lint`/`typecheck`만 강제.

### 논의 필요

- 빌드 실패를 PR 단계가 아닌 CD 단계에서 받는 운영 리스크(무중단 배포 전제).

### 선택지

- A. 훅에서 `build` 완전 제거하고 CD만 빌드 검증
- B. `apps/web` 변경 시에만 훅에 `build` 추가

### 추천

- A. 훅에서는 제거, CD에서 빌드 게이트 유지

### 사용자 방향

- 기본안(A) 채택
