# 공통 헤더 + 커뮤니티 화면 재구성 백로그

## 목적

홈(`/`), 타자 서비스, 플래시카드 서비스, 커뮤니티(`/community`)가 같은 공통 헤더를 사용하도록 정리하고, `/community`를 소개 페이지가 아닌 실시간 채팅 + 게시글 목록 중심의 실제 커뮤니티 피드 화면으로 재구성한다.

## 범위

- 공통 헤더 적용 범위: 홈, 타자 서비스, 플래시카드 서비스, 커뮤니티.
- 헤더 구성: YEON 로고, 타자/플래시카드/커뮤니티 메뉴, BGM pill, 설정 아이콘 버튼, 내정보 아이콘 버튼.
- 아이콘 정책: 새 이미지/SVG 추가 금지, `react-icons` 사용 금지, 기존 아이콘 우선 재사용, 신규 통일 필요 시 `lucide-react` 사용.
- 커뮤니티 본문: 작은 실시간 채팅, 카테고리 필터, 글쓰기 펼침 폼, 게시글 카드 목록, 인기 글, 최근 댓글, 게시글 상세/댓글 작성·삭제 흐름.
- 제외: 알림, 검색, 커뮤니티 전용 큰 제목, 긴 설명 문구, 마케팅/소개 페이지형 카피.

## 1차 — 현재 구현 조사와 공통 헤더 source of truth 결정

### 작업내용

- 홈, 타자 서비스, 플래시카드 서비스, 커뮤니티에서 현재 사용 중인 헤더 컴포넌트와 BGM/설정/프로필 버튼 구현 위치를 조사한다.
- 기존 프로젝트 아이콘 컴포넌트와 `lucide-react` 의존성 사용 여부를 확인한다.
- 공통 헤더를 새로 만들지, 기존 헤더를 승격할지 결정하고 적용 대상 경로별 영향 범위를 정리한다.
- 헤더에서 제거해야 하는 알림/검색/커뮤니티 전용 제목/긴 설명 문구가 실제 화면에 있는지 확인한다.

### 논의 필요

- 공통 헤더를 모든 서비스에 즉시 완전 동일 적용할지, 기존 서비스 헤더의 내부 동작은 유지하면서 외형과 우측 유틸만 먼저 맞출지 결정이 필요하다.
- 타자/플래시카드 서비스의 기존 설정/BGM 상태 source of truth를 헤더 공통 컴포넌트가 직접 소유해도 되는지 확인이 필요하다.

### 선택지

- 옵션 A: 기존 제품 헤더 래퍼를 확장해 공통 구조와 우측 유틸만 통일하고, 각 서비스의 내부 상태 연결은 기존 구현을 유지한다.
- 옵션 B: 새 `CommonHeader`를 만들고 4개 화면에서 모두 동일 컴포넌트를 사용한다.
- 옵션 C: `/community`에 먼저 공통 헤더를 적용하고, 홈/타자/플래시카드는 후속 차수로 미룬다.

### 추천

- 옵션 A. 기존 동작을 덜 흔들면서 사용자가 요구한 공통 헤더 외형, 아이콘 정책, 메뉴 구조를 가장 작게 통일할 수 있다.

### 사용자 방향

- 추천 기준으로 진행한다.

## 2차 — 공통 헤더 UI 적용

### 작업내용

- 좌측 YEON 로고를 홈 링크로 고정한다.
- 메뉴는 `타자`, `플래시카드`, `커뮤니티` 순서로 배치하고 각 서비스 경로로 연결한다.
- BGM 버튼은 기존 pill 형태를 유지하고 `BGM Kevin MacLeod` 텍스트와 상태 아이콘을 함께 표시한다.
- BGM 꺼짐은 `VolumeX`, 켜짐은 `Volume2`를 사용한다.
- 설정 버튼은 텍스트 없이 `Settings` 아이콘 버튼으로 표시한다.
- 내정보 버튼은 텍스트 `내정보` 없이 `CircleUserRound` 우선, 필요 시 `UserRound`로 표시한다.
- 새 이미지 파일이나 직접 작성 SVG를 추가하지 않는다.
- `react-icons`를 새로 쓰지 않고, 새 통일이 필요할 때만 `lucide-react` 기준으로 맞춘다.

### 논의 필요

- BGM 버튼의 실제 재생/음소거 상태가 서비스별로 다르면 헤더 표시를 어떤 상태값에 연결할지 확인해야 한다.
- 내정보 버튼 클릭 시 기존 프로필 메뉴/인증 모달/설정 패널 중 어느 흐름을 유지할지 화면별 현재 구현을 기준으로 결정해야 한다.

### 선택지

- 옵션 A: 공통 헤더는 표시와 트리거만 담당하고, BGM/설정/내정보 동작은 각 화면에서 주입한다.
- 옵션 B: 공통 헤더가 모든 유틸 상태와 모달을 직접 소유한다.
- 옵션 C: 아이콘/링크만 먼저 통일하고 BGM/설정/내정보 동작 연결은 별도 후속 작업으로 둔다.

### 추천

- 옵션 A. 상태 source of truth가 흩어지는 것을 막고, 기존 서비스별 동작을 유지하면서 헤더 외형을 통일할 수 있다.

### 사용자 방향

- 추천 기준으로 진행한다.

## 3차 — `/community` 기본 피드 레이아웃 재구성

### 작업내용

- `/community` 상단에 공통 헤더를 적용하고 커뮤니티 전용 큰 제목/소개 문구를 제거한다.
- 헤더 아래 중앙 상단에 낮고 촘촘한 실시간 채팅 박스를 배치한다.
- 채팅 박스는 제목, 접속자 수, 최근 메시지 2~3개, 닉네임 입력, 메시지 입력, 전송 버튼만 포함한다.
- 게시글 영역은 카테고리 필터, `글쓰기` 버튼, 게시글 카드 목록이 즉시 보이도록 구성한다.
- 카테고리는 `전체`, `잡담`, `타자친구 모집`, `카드친구 모집`, `관리자에게 아무말/조언` 순서와 문구를 그대로 사용한다.
- 우측 사이드에 `인기 글`과 `최근 댓글`을 작고 촘촘한 리스트로 배치한다.

### 논의 필요

- 모바일 화면에서 우측 사이드 영역을 게시글 아래로 내릴지, 접기 섹션으로 둘지 결정이 필요하다.
- 실시간 채팅 접속자 수가 실제 데이터가 없을 때 임시 표시를 숨길지, 기존 API 값만 사용할지 결정이 필요하다.

### 선택지

- 옵션 A: 데스크톱은 본문+우측 사이드 2열, 모바일은 채팅→필터/글쓰기→게시글→인기 글→최근 댓글 순서 단일 열로 배치한다.
- 옵션 B: 모든 화면에서 단일 열로 단순화한다.
- 옵션 C: 데스크톱 우측 사이드를 고정하고 모바일에서는 숨긴다.

### 추천

- 옵션 A. 게시글 중심 밀도를 유지하면서 모바일에서도 정보 손실을 만들지 않는다.

### 사용자 방향

- 추천 기준으로 진행한다.

## 4차 — 글쓰기 펼침 폼과 게시글 카드 정보 밀도 정리

### 작업내용

- `글쓰기` 폼은 기본 상태에서 숨기고, 버튼 클릭 시 게시글 목록 위에 펼친다.
- 글쓰기 필드는 카테고리, 제목, 내용, 닉네임, 비밀번호, 등록, 취소로 구성한다.
- 게시글 카드는 제목, 본문 미리보기, 카테고리 배지, 닉네임, 작성 시간, 댓글 수를 반드시 표시한다.
- 글쓰기 폼이 열려도 게시글 목록이 화면의 핵심으로 남도록 폼 높이와 여백을 과하게 키우지 않는다.
- 빈 목록 문구 `아직 글이 없습니다. 첫 글을 남겨보세요.`는 노출하지 않는다.

### 논의 필요

- 빈 목록일 때 아무 문구도 숨길지, 아주 짧은 운영형 대체 문구를 둘지 결정이 필요하다. 단, 사용자가 금지한 문구는 사용하지 않는다.
- 글쓰기 폼 입력 검증 실패 메시지의 톤을 기존 커뮤니티 API 오류와 맞출지 확인이 필요하다.

### 선택지

- 옵션 A: 빈 목록은 카드 영역을 비워 두고, 개발/운영 정책 문구는 화면에 노출하지 않는다.
- 옵션 B: 금지 문구와 다른 짧은 빈 상태 문구를 추가한다.
- 옵션 C: 샘플 게시글을 UI에 하드코딩한다.

### 추천

- 옵션 A. 사용자가 명시한 “설명 페이지가 아니라 피드” 방향과 금지 문구 조건을 가장 안전하게 지킨다.

### 사용자 방향

- 추천 기준으로 진행한다.

## 5차 — 게시글 상세와 댓글 작성/삭제 흐름 확인

### 작업내용

- 게시글 카드 클릭 시 `/community/posts/{postId}` 상세 이동 흐름을 유지하거나 추가한다.
- 상세 화면은 제목, 카테고리, 작성자 닉네임, 작성 시간, 본문, 수정/삭제 버튼, 댓글 목록, 댓글 작성 폼을 제공한다.
- 댓글 작성 필드는 닉네임, 비밀번호, 댓글 내용, 등록 버튼으로 구성한다.
- 댓글 삭제는 제공하고, 댓글 수정/대댓글/좋아요는 추가하지 않는다.
- 댓글 수정 미제공 문구는 화면에 노출하지 않는다.

### 논의 필요

- 현재 상세 라우트가 이미 존재하면 화면 정의에 맞춰 UI만 조정할지, 목록 화면 우선 후 별도 PR로 나눌지 결정이 필요하다.
- 게시글 수정/삭제 비밀번호 확인 UI가 기존 구현에 있으면 그대로 유지할지, 이번 화면 정의에 맞춰 재배치할지 확인이 필요하다.

### 선택지

- 옵션 A: 상세/댓글은 기존 기능을 보존하고 화면 정의에서 벗어난 문구와 과한 설명만 제거한다.
- 옵션 B: 상세 페이지까지 같은 PR에서 전면 재구성한다.
- 옵션 C: 상세 페이지는 링크만 확인하고 이번 차수에서는 목록 화면까지만 구현한다.

### 추천

- 옵션 A. 신규 기능을 임의로 추가하지 말라는 조건을 지키면서 상세/댓글 흐름의 사용자 경험만 맞춘다.

### 사용자 방향

- 추천 기준으로 진행한다.

## 6차 — 검증, main 반영, 릴리즈 판단

### 작업내용

- 금지 문구가 화면 코드에 남아 노출되지 않는지 검색한다.
- 새 이미지/SVG 파일 추가가 없는지 확인한다.
- `react-icons` 신규 사용이 없는지 확인한다.
- 관련 lint/typecheck/build를 실행한다. 웹 화면 변경이므로 `pnpm --filter @yeon/web build`를 포함한다.
- 변경 파일만 선별 stage 후 한국어 커밋 메시지로 커밋한다.
- 브랜치 push → PR(main) 생성 → merge까지 진행한다.
- 변경 성격에 따라 release 필요 여부와 PATCH/MINOR 판단을 남긴다.

### 논의 필요

- 화면 재구성은 사용자 가시 UI 변경이므로 release가 필요할 가능성이 높다. 버전 bump 수준은 구현 범위 확정 후 판단한다.
- CI가 장시간 지연되거나 외부 배포 확인이 막히면 자동 머지/수동 확인 중 어느 선에서 멈출지 결정이 필요하다.

### 선택지

- 옵션 A: 구현 완료 후 검증이 통과하면 PR(main) merge까지 즉시 완료하고, release 필요 시 PATCH로 처리한다.
- 옵션 B: PR 생성까지만 하고 사용자 확인 후 merge한다.
- 옵션 C: 커뮤니티 목록 화면과 공통 헤더를 PR로 분리한다.

### 추천

- 옵션 A. 이 저장소의 main-only 운영 정책과 “작업하고 나서 메인에 반영” 요청에 맞다.

### 사용자 방향

- 작업 후 main에 반영한다.

## 화면 노출 금지 문구 체크리스트

아래 문구는 사용자 화면에 노출하지 않는다.

- `커뮤니티 공간`
- `실시간 채팅과 글/댓글이 함께 보이는 공간입니다`
- `글 CRUD`
- `닉네임+비밀번호로 처리합니다`
- `로그인하지 않아도 닉네임과 비밀번호를 입력하면`
- `아직 글이 없습니다. 첫 글을 남겨보세요.`

## 완료 기준

- 홈/타자/플래시카드/커뮤니티가 동일한 공통 헤더 구조를 사용한다.
- 헤더 아이콘은 기존 아이콘 재사용 또는 `lucide-react` 기준으로 통일되어 있다.
- `/community` 첫 화면에서 작은 실시간 채팅과 게시글 목록이 먼저 보인다.
- 글쓰기 폼은 기본 숨김이고 버튼 클릭 시 목록 위에 펼쳐진다.
- 우측에 인기 글/최근 댓글 리스트가 표시된다.
- 금지 문구, 새 이미지/SVG, 신규 `react-icons` 사용이 없다.
- 웹 빌드와 관련 검증이 통과한다.
- PR(main) merge 및 release 필요 여부 판단이 완료된다.
