# 타자방 방 종류 고정 UX 개선

## 1차

### 작업내용

- 타자방 생성 시 `일반 타자방`과 `점령전 방` 중 하나를 선택하게 한다.
- 생성된 방은 방 종류를 source of truth로 유지하고 대기방 안에서는 해당 종류에 맞는 액션만 노출한다.
- 일반 타자방은 `시작하기`/`준비하기` 흐름만 제공한다.
- 점령전 방은 `점령전 입장` 흐름만 제공한다.

### 논의 필요

- 방 생성 후 방 종류 변경을 허용할지 여부.

### 선택지

- 선택지 A: 생성 후 변경 불가. 방 종류는 방 생성 계약에 고정한다.
- 선택지 B: 대기방 설정에서 방 종류를 바꿀 수 있게 한다.

### 추천

- 선택지 A. 한 방 안에서 점령전과 일반 시작이 동시에 가능한 UX가 문제였으므로 방 종류는 생성 시 고정하는 것이 명확하다.

### 사용자 방향

- 점령전으로 방을 파거나, 그냥 모드로 시작해서 방을 파는 구조로 정리한다.

## 2차

### 작업내용

- 타자방 생성 모달과 대기실 헤더에서 방 제목을 방 종류보다 먼저 보이게 한다.
- 점령전 방의 팀 이동을 실제 서버 상태 변경으로 연결한다.
- 점령전 대기실 운영 전용 admin 페이지를 만들고, 지정 방에 유효한 대기 참가자를 넣을 수 있게 한다.

### 논의 필요

- admin이 넣는 참가자를 실제 로그인 DB 사용자로 만들지, 레이스 룸 내부의 유효한 연습 참가자로 만들지 결정해야 한다.

### 선택지

- 선택지 A: Spring DB에 실제 계정을 생성한 뒤 race-server에 참가자로 연결한다.
- 선택지 B: race-server 내부에 유효한 participant 필드를 가진 연습 참가자를 생성하되 경험치 적립 대상에서는 제외한다.

### 추천

- 선택지 B. 현재 요구는 팀 대기실 인원 채우기이며, 계정/인증/영속 DB 생성은 Spring 소유라 이 작업의 범위를 넘는다. race-server 안에서 유효한 참가자 데이터를 생성하되 `verifiedUserId`를 비워 경험치 적립과 실제 계정 보상에는 영향이 없게 한다.

### 사용자 방향

- 방 제목이 방 종류보다 위에 있어야 한다.
- 팀 이동이 실제로 동작해야 한다.
- admin 페이지는 팀 대기실에 문제없는 데이터를 가진 유효한 참가자를 넣는 역할만 수행한다.
