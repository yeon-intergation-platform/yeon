# typing room stability random seed

## 1차

- 작업내용: 타자방 생성 직후 `room "..." not found`로 연결이 끊기는 문제를 재현/추적하고, 방장이 일시적으로 leave 되어도 방이 즉시 폐기되지 않도록 안정화한다.
- 논의 필요: 방장 연결이 끊겼을 때 즉시 방을 닫을지, 짧은 재접속 유예 시간을 둘지 결정이 필요하다.
- 선택지:
  - A. 방장 leave 즉시 방 종료를 유지한다.
  - B. 방장 leave 시 짧은 유예 후 여전히 참가자가 없을 때만 종료한다.
- 추천: B. 로컬 dev/React 재마운트/네트워크 찰나 끊김에서도 방 생성 직후 room id가 사라지지 않아야 한다.
- 사용자 방향: B

## 2차

- 작업내용: 타자방 게임 시작 시 같은 덱/방이라도 매 라운드 시작마다 race seed를 새로 뽑아 참가자 모두에게 동일한 무작위 문장을 전달한다.
- 논의 필요: seed 생성 권한을 서버가 완전히 가질지, 기존 클라이언트/API seed 생성 경로를 유지하며 서버가 여러 seed 후보 중 선택할지 결정이 필요하다.
- 선택지:
  - A. 서버 fallback 문장만 무작위화한다.
  - B. 클라이언트가 방 생성 시 여러 seed 후보를 넘기고, 서버가 게임 시작마다 후보 중 하나를 선택한다.
- 추천: B. 기존 덱/API seed token 검증 구조를 유지하면서도 모든 참가자에게 같은 무작위 문장을 보장할 수 있다.
- 사용자 방향: B

## 3차

- 작업내용: 바로 레이스 결과 화면의 `다시 레이스`가 기존 연결 옵션의 race seed를 재사용하지 않고, 재입장 전에 선택 덱 기준 새 race seed를 다시 resolve하게 한다.
- 논의 필요: 재시작을 같은 방 재사용으로 볼지, 새 방/새 seed 재입장으로 볼지 사용자 경험 기준이 필요하다.
- 선택지:
  - A. 기존 방을 재입장하되 seed를 유지한다.
  - B. 새 seed를 만든 뒤 quick room을 다시 생성/입장한다.
- 추천: B. `다시 레이스`는 사용자가 새 판으로 인식하므로 같은 덱 안에서도 문장이 바뀌어야 한다.
- 사용자 방향: B
