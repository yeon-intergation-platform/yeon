# 카드 서비스 화면 품질 루프

## 목표

- `card-service`만 범위로 고정해 실제 사용자 화면 품질을 점검한다.
- Playwright로 첫 진입, 게스트/로그인 분기, 덱 생성, 수정, 삭제, 새로고침, 모바일, 실패 입력, 느린 네트워크, 빈/에러/로딩 상태를 확인한다.
- 스크린샷 기준으로 다음 행동, 중복 UI, CTA 우선순위, 모바일 안정성, 빈/에러/로딩 안내, 기존 디자인 톤 유지 여부를 평가한다.
- P0/P1 중 가장 작고 독립적인 묶음만 수정하고 재발 방지 장치를 남긴다.

## 1차 - main 기준 분리와 근거 로드

### 작업내용

- `origin/main` 기준 `qa/card-service-visual-loop-20260619` 브랜치에서 진행한다.
- 카드 서비스 규칙, 디자인 작업 절차, 카드 서비스 컨텍스트, Universal UI parity registry를 확인한다.
- 수정 전 현재 화면과 source of truth를 먼저 확인한다.

### 논의 필요

- 없음. 사용자가 루프 절차를 명시했다.

### 선택지

- A. 기존 i18n 작업 브랜치에서 이어서 카드 UI QA를 진행한다.
- B. 깨끗한 작업 워크트리에서 `origin/main` 기준 새 브랜치로 분리한다.

### 추천

- B. 다른 작업과 섞이지 않고 PR(main)로 바로 반영할 수 있다.

### 사용자 방향

- 추천 기준으로 진행한다.

## 2차 - Playwright 화면 감사

### 작업내용

- 카드 홈, 덱 목록, 덱 생성 모달, 덱 상세, 학습 화면을 실제 브라우저로 확인한다.
- 데스크톱과 모바일 viewport 스크린샷을 저장한다.
- 로딩, 빈 데이터, 실패 입력, 새로고침, 느린 네트워크 상태를 확인한다.

### 논의 필요

- 로그인 세션이 없거나 외부 인증이 필요한 흐름은 게스트 경로를 우선하고, 인증 전용 결함은 재현 가능 범위와 리스크를 분리해 기록한다.

### 선택지

- A. 코드만 읽고 UI를 추정한다.
- B. Playwright 실제 스크린샷과 동작 결과를 근거로 판단한다.

### 추천

- B. 이번 목표는 화면 품질 루프이므로 스크린샷 증거가 필수다.

### 사용자 방향

- 추천 기준으로 진행한다.

## 3차 - 최소 수정과 재발 방지

### 작업내용

- 발견한 문제를 P0/P1/P2로 분류한다.
- 한 사이클에서는 P0/P1 중 가장 작은 독립 묶음만 수정한다.
- 수정마다 Playwright smoke, screenshot check, unit/typecheck/lint 중 하나 이상을 재발 방지 장치로 추가한다.

### 논의 필요

- DB 마이그레이션, 인증/권한 정책 변경, 디자인 시스템 자체 변경이 필요하면 즉시 중단한다.

### 선택지

- A. 화면을 크게 재설계한다.
- B. 기존 톤 안에서 레이아웃, 정보 구조, CTA, 상태 안내만 작게 개선한다.

### 추천

- B. 카드 서비스 SSOT가 흰 배경, 검정 CTA, 회색 보더의 생산성 UI를 요구한다.

### 사용자 방향

- 추천 기준으로 진행한다.

## 4차 - 검증, PR, main 반영

### 작업내용

- 관련 lint/typecheck/test/Playwright 검증을 실행한다.
- 수정 전/후 스크린샷과 개선 내용을 작업 로그에 남긴다.
- PR(main)을 생성하고 merge한다. CI 실패 시 로그를 근거로 다음 사이클에서 최소 수정한다.

### 논의 필요

- GitHub API rate limit이나 비동기 Actions 상태처럼 외부 상태가 막히면 URL과 남은 리스크를 남긴다.

### 선택지

- A. 로컬 수정만 하고 멈춘다.
- B. 프로젝트 정책에 따라 PR(main) merge까지 진행한다.

### 추천

- B. AGENTS와 사용자 절차 모두 main 반영까지 요구한다.

### 사용자 방향

- 추천 기준으로 진행한다.
