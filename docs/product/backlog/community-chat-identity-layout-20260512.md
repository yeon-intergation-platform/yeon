# 커뮤니티 채팅/작성자 정보 배치 개선 백로그

## 1차수

### 작업내용

- `/community` 상단 실시간 채팅 영역의 가로 폭을 넓혀 페이지 상단 영역을 더 많이 사용하게 한다.
- 글/댓글 CRUD에 쓰는 비회원 닉네임/비밀번호 입력 영역을 실시간 채팅 카드 오른쪽에 배치한다.
- 기존 피드 헤더 아래에 있던 닉네임/비밀번호 입력 중복 위치는 제거한다.

### 논의 필요

- 실시간 채팅 전송용 닉네임과 글/댓글 CRUD용 닉네임/비밀번호를 완전히 통합할지는 별도 정책 결정이 필요하다.

### 선택지

1. 닉네임/비밀번호 입력을 기존 피드 헤더 아래에 유지한다.
2. 닉네임/비밀번호 입력을 상단 실시간 채팅 카드 오른쪽으로 이동한다.

### 추천

- 2안. 커뮤니티 진입 시 비회원 작성자 정보를 먼저 설정하고 채팅/피드 흐름으로 바로 이어지게 만든다.

### 사용자 방향

- 커뮤니티에서 닉네임/비밀번호 위치를 실시간 채팅 카드 오른쪽에 둔다.
- 채팅 가로 넓이를 더 넓힌다.

## 2차수

### 작업내용

- 커뮤니티 상단 실시간 채팅에서는 닉네임 입력칸을 제거한다.
- 카드/타자 서비스 내부 실시간 채팅은 닉네임 입력칸을 유지한다.
- 비회원 기본 닉네임은 `익명이`로 둔다.
- `/profile` 내정보 화면에서는 이름을 제거하고 이메일/로그인 방식/최근 로그인만 보여준다.

### 논의 필요

- 기존 브라우저에 저장된 과거 채팅 닉네임까지 강제로 `익명이`로 마이그레이션할지는 별도 결정이 필요하다.

### 선택지

1. 모든 실시간 채팅에서 닉네임칸을 제거한다.
2. 커뮤니티 상단 채팅에서만 닉네임칸을 제거하고 카드/타자 내부 채팅은 유지한다.

### 추천

- 2안. 커뮤니티는 오른쪽 작성자 정보가 있으므로 중복 입력을 줄이고, 카드/타자 내부 위젯은 독립 사용성이 필요하다.

### 사용자 방향

- 커뮤니티 실시간 채팅에서는 닉네임칸이 필요 없다.
- 카드서비스나 타자서비스 실시간 채팅에서는 닉네임칸이 있어야 한다.
- 기본 닉네임은 `익명이`로 한다.
- 내정보에서는 이름을 빼고 이메일/로그인 방식/최근 로그인만 보여준다.

## 3차수

### 작업내용

- 타자연습 관리자용 `캐릭터 프레임 설정` 링크를 헤더 바깥 별도 줄에서 제거한다.
- 해당 링크는 타자연습 공통 헤더 오른쪽 컨트롤 영역에 넣어 헤더 높이를 일관되게 유지한다.

### 논의 필요

- 관리자 링크를 텍스트 버튼으로 유지할지 아이콘 버튼으로 축약할지는 추후 관리자 기능이 늘어날 때 결정한다.

### 선택지

1. 별도 상단 줄을 유지한다.
2. 헤더 오른쪽 컨트롤에 넣는다.

### 추천

- 2안. 헤더 크기와 화면 시작 위치가 서비스별로 일관된다.

### 사용자 방향

- `캐릭터 프레임 설정` 링크가 별도 div 공간 전체를 차지하지 않게 하고 공통 헤더 안에 둔다.

## 4차수

### 작업내용

- 타자방 대기 화면에서 방설정, 참여자, 채팅 카드를 데스크톱 기준 한 행에 배치한다.
- 대기 화면의 상단 여백과 카드 높이를 줄여 스크롤 없이 주요 정보가 뷰포트 안에 들어오게 한다.
- 캐릭터/채팅/설정 영역의 세로 크기를 줄이되 기존 방 생성/설정/채팅 동작은 유지한다.

### 논의 필요

- 모바일에서는 한 행 고정이 현실적으로 좁으므로 기존처럼 세로 적층을 유지한다.

### 선택지

1. 기존 2열 레이아웃을 유지한다.
2. 데스크톱에서 방설정/참여자/채팅 3열 레이아웃으로 압축한다.

### 추천

- 2안. 타자방 대기 화면의 핵심 제어/상태/대화가 한눈에 들어온다.

### 사용자 방향

- 타자방에서 방설정/참여자/채팅카드가 한 행에 있어야 한다.
- 스크롤 없이 뷰포트 내에 보여야 한다.

## 5차수

### 작업내용

- 카드/타자 서비스 내부 compact 실시간 채팅 위젯의 접기/열기 UI를 텍스트 버튼에서 아이콘 버튼으로 바꾼다.
- compact 위젯에서는 메시지 개수와 `채팅 위젯이 접혔습니다.` 문구를 제거한다.
- framer-motion으로 접힘/펼침 전환을 부드럽게 처리한다.

### 논의 필요

- compact 위젯의 아이콘 종류는 현재 메시지/닫기 의미가 분명한 Lucide 아이콘으로 둔다.

### 선택지

1. 기존 텍스트 버튼과 개수 표시를 유지한다.
2. 아이콘 버튼 중심의 compact 헤더로 단순화한다.

### 추천

- 2안. 좁은 floating 카드에서 텍스트 줄바꿈과 깨진 느낌을 줄인다.

### 사용자 방향

- `열기`, `3개`, `채팅 위젯이 접혔습니다.`는 필요 없다.
- Lucide 아이콘과 framer-motion으로 깔끔하게 고친다.

## 6차수

### 작업내용

- 타자방 생성 직후 URL을 `/typing-service/rooms/{roomId}`로 교체해 새로고침 시 새 방을 만들지 않게 한다.
- race-server에서 참가자 식별자를 Colyseus sessionId가 아니라 클라이언트의 안정적인 `playerId` 기준으로 관리한다.
- 새로고침/일시 연결 끊김은 즉시 퇴장 처리하지 않고 짧은 유예 시간 동안 같은 참가자로 재연결되게 한다.
- 방장이 새로고침해도 방이 바로 닫히지 않게 한다.
- 1판 종료 후 결과 표시 시간을 둔 뒤 같은 타자방을 `waiting` 상태로 되돌려 참가자들이 안전하게 로비로 복귀하게 한다.

### 논의 필요

- 재연결 유예 시간과 결과 표시 시간은 운영 피드백 후 조정 가능하다.

### 선택지

1. 클라이언트만 재시도하게 둔다.
2. 서버 상태 모델을 안정 식별자/재연결 유예/라운드 후 로비 복귀로 정리한다.

### 추천

- 2안. 새로고침, 채팅 유실, 게임 종료 후 복귀 에러가 모두 같은 상태관리 문제라 서버와 URL 양쪽을 함께 고쳐야 한다.

### 사용자 방향

- 새로고침으로 새 타자방이 만들어지면 안 된다.
- 새로고침 시 채팅이 사라지면 안 된다.
- 1판 종료 후 같은 방으로 안전하게 돌아와야 한다.

## 7차수

### 작업내용

- compact 채팅 위젯의 접힌 상태는 원형 아이콘만 보이게 한다.
- compact 위젯의 `실시간 채팅` 텍스트 헤더를 제거한다.
- 펼친 compact 채팅 위젯의 가로 폭을 기존 대비 약 2배로 넓힌다.
- 메시지 전송 후 입력창 포커스가 유지되도록 한다.

### 논의 필요

- 펼친 폭은 모바일 뷰포트에서는 화면 폭을 넘지 않도록 `100vw` 기준으로 제한한다.

### 선택지

1. 헤더 텍스트를 유지하고 버튼만 줄인다.
2. compact 상태에서는 아이콘 중심 floating 위젯으로 재구성한다.

### 추천

- 2안. 카드/타자 화면 안의 floating 위젯은 좁은 헤더보다 아이콘 중심이 덜 깨져 보인다.

### 사용자 방향

- `실시간 채팅` 텍스트와 헤더 div를 없애고 아이콘만 둔다.
- 아이콘 크기를 키운다.
- 펼쳤을 때 가로 폭은 현재보다 2배 넓힌다.
- 메시지 전송 후 입력 활성 상태가 풀리는 문제를 해결한다.

## 8차 - 타자연습 홈 2열 레이아웃 보강

- 작업내용: `/typing-service` 홈에서 `내 프로필`과 `오늘의 시작` CTA 3개를 하나의 2열 패널로 묶어 첫 진입 뷰포트 안에 버튼이 함께 보이게 한다.
- 논의 필요: 모바일에서는 1열 유지가 필요하다.
- 선택지: 1) 기존 세로형 유지 2) 데스크톱 2열/모바일 1열 반응형 전환
- 추천: 2) 데스크톱 2열/모바일 1열 반응형 전환
- 사용자 방향: 내 프로필/오늘의 시작 2열 패널, CTA는 제목과 설명이 있는 큰 카드형 버튼으로 구성

## 9차 - compact 채팅 상태 표시 축약

- 작업내용: compact 채팅을 접었을 때는 아이콘만 보이고, 펼쳤을 때만 제목과 접속 수를 표시하되 접속 수는 문구 없이 초록색 숫자 pill로 표시한다.
- 논의 필요: 현재 HTTP 폴링 기반이므로 실제 소켓 접속자 수가 아니라 최근 채팅 작성자 기준 근사값이다.
- 선택지: 1) 텍스트 설명 유지 2) 초록색 숫자 pill만 표시 3) 별도 presence API 추가
- 추천: 2) 이번 UI 수정에서는 초록색 숫자 pill만 표시하고, 정확한 실시간 presence는 별도 API 작업으로 분리
- 사용자 방향: 말로 표현하지 말고 초록색 버튼으로 표시

## 10차 - 사이트 접속자 presence 기준 전환

- 작업내용: 채팅의 접속자 수를 최근 채팅 작성자 수가 아니라 사이트 접속 세션 heartbeat 기준으로 계산한다.
- 논의 필요: 다중 서버/서버리스 환경에서는 프로세스 메모리 기반 presence가 인스턴스별로 나뉠 수 있다.
- 선택지: 1) 최근 채팅 작성자 기반 유지 2) 앱 서버 메모리 heartbeat 3) Redis 등 공유 presence 저장소
- 추천: 2) 현재 운영 구조에서는 앱 서버 heartbeat로 즉시 반영하고, 다중 인스턴스 확장 시 3)으로 승격
- 사용자 방향: 서비스에 들어와 있는 사람 수가 실시간 인원으로 집계되어야 함
