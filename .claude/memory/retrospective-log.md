# Retrospective Log

## 2026-06-19 UI 디자인 변경 스크린샷 증거

- 증상: UI/디자인 변경을 완료했다고 보고해도 실제 화면의 어느 위치가 어떻게 바뀌었는지 나중에 추적하기 어려웠다.
- 실제 원인: Playwright 확인과 스크린샷 저장은 작업자 판단에 맡겨져 있었고, 작업 로그/PR에 증거 경로를 남기는 완료 조건이 명시되어 있지 않았다.
- 더 이른 시점에 잡을 수 있었던 신호: 디자인 피드백이 반복될 때마다 전후 화면 증거가 작업 산출물로 남아 있는지 먼저 확인했어야 했다.
- 다음부터 적용할 규칙: UI/디자인이 보이는 변경은 `docs/guides/design-screenshot-evidence.md`에 따라 before/after 또는 after 스크린샷을 남기고, 작업 로그와 PR에 경로와 시각적 변경점을 기록한다.

## 2026-04-07 소셜 로그인 round 6

- 증상: 구글 로그인 화면을 열어둔 상태에서 카카오 로그인도 시작하거나, 같은 공급자 로그인을 연속으로 두 번 시작하면 첫 번째 콜백이 `invalid_state`로 실패할 수 있었다.
- 실제 원인: OAuth pending 상태를 단일 서명 쿠키 1개로만 관리해, 새 로그인 시작이 이전 `state`를 덮어썼다.
- 더 이른 시점에 잡을 수 있었던 신호: state 검증은 들어가 있었지만, "동시에 두 개 이상의 로그인 시도가 있을 수 있는가"를 플로우 설계 단계에서 따로 점검하지 않았다.
- 다음부터 적용할 규칙: OAuth `state`는 콜백 1회성 값으로만 보지 말고, 같은 브라우저에서 여러 로그인 시도가 겹칠 수 있다는 전제로 다중 pending 상태 또는 공급자별 격리를 먼저 검토한다.
