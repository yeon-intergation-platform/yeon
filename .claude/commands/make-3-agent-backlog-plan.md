# make-3-agent-backlog-plan

## Purpose

이 명령은 **코드 수정 없이 작업 백로그만 생성**한다.
`/make-3-agent-backlog-plan ...` 호출 시 `yeon-2`, `yeon-3`, `yeon-4`의 작업 로그(및 필요시 백로그 파일)를 생성해 둔다.
핵심은 총 3개 lane 플랜을 준비할 수 있게 해두되, 작은 단일 작업은 **한 lane에만** 쌓는다.

## 워크트리 기본 정책

- 기본적으로는 **순차 처리(1 lane)**를 우선한다.
- 단일 요청은 기본적으로 1개의 lane만 생성한다.
- 여러 lane 분할은 아래의 조건이 모두 충족될 때만 허용한다.
  1. 매우 큰 범위의 작업으로 **독립적인 실행 경로**가 분명한 경우
  2. 서로 다른 결과물이 겹치지 않으며, 동시 시작해도 영향 범위가 격리되는 경우
  3. 실패해도 다른 lane의 진행이 무너지지 않는 경우
- 요청자가 `yeon-2`, `yeon-3`, `yeon-4` 분할을 직접 명시했거나,
  요청 내용 자체가 명확히 2~3개 독립 작업으로만 구성되는 경우에만 2~3 lane을 생성한다.
- 문서/정책 정리와 개발, 또는 회귀 대응을 기본적으로 같은 요청에서 병렬로 나누지 않는다.
  (컨텍스트 유지가 중요한 작업은 순차적으로 한 사람이 담당한다.)

## 분할 가드

- yeon-3를 “문서 정리 전담”, yeon-4를 “회귀/QA 전담”으로 강제 분할하는 방식은 기본 운영 방식이 아니다.
- `요청 → 실행` 흐름이 선형적이면 lane 분할하지 말고 1차 백로그로 유지한다.
- 예외적으로 매우 큰 작업에서 아래를 만족하면 병렬 분할을 허용한다.
  - 서로 다른 서비스/코드 경계가 명확히 분리됨
  - 의존성/선행조건이 없음
  - 병렬 수행 후 결과 병합 지점이 명확함

## 실행 차단/가이드 규칙

- `/make-3-agent-backlog-plan`은 **백로그 생성 전용**이며, 실제 코드 작업 실행을 수행하지 않는다.
- 사용자가 `yeon에서 작업해`, `바로 실행`, `개발해`, `구현해`처럼 실수행 동작을 지시하면 다음 경고를 먼저 출력해야 한다.

```text
경고: /make-3-agent-backlog-plan은 실행 지시를 대신하지 않습니다.
백로그는 여기서 쌓고, 실제 작업은 yeon-2/3/4에서 수행하세요.
```

- 동시에, 아래 작업 실행 가이드를 반드시 함께 출력한다.

````text
권장: 작업 실행은 3개 워크트리에 에이전트 CLI 세션을 열고 분배한다.
1) yeon-2: `cd ~/coding_stuffs/yeon-2 && codex`
2) yeon-3: `cd ~/coding_stuffs/yeon-3 && codex`
3) yeon-4: `cd ~/coding_stuffs/yeon-4 && codex`

각 세션에 이번 요청의 작업내용(요구사항/범위/완료 기준/제약사항)을 동일하게 전달하고 실행한다.
로컬 테스트 실행은 `yeon`(메인 워크트리)에서는 오직 개발자만 수행한다.
에이전트는 각자의 세션(yeon-2/yeon-3/yeon-4)에서 테스트를 분담/진행한다.
각 세션은 작업 시작 시점에 `origin/main` 기준으로 클린 브랜치를 생성해서 시작해야 한다.

예시:
```text
cd ~/coding_stuffs/yeon-2
git fetch origin
git checkout main
git reset --hard origin/main
git switch -c <issue-or-task-name>
````

````

## 입력 규칙

- 기본 호출: `/make-3-agent-backlog-plan <요청 요약 및 할 일>`
- lane 강제: `yeon-2: ...`, `yeon-3: ...`, `yeon-4: ...` 형태로 문장 앞에 접두어 가능
- 다중 항목: `;` 또는 줄바꿈으로 작업 분리 가능 (`요청을 2~3개 독립 플랜으로 고의 분할한 경우만`).
- lane 자동 추정 없이, **명시된 접두어가 최우선**이다.
- 보수적 분할 원칙: 기본은 단일 lane, 고도의 병렬 필요성이 확실한 경우에만 2~3 lane.

## 실행 결과물 (반드시 생성)

요청마다 lane별로 필요한 개수만큼(기본 1개, 최대 3개) 작업 로그를 만든다.

1. `ai-log/{person}/YYYY-MM-DD/`
   - 파일명 형식:
     - `N-작업-codex_{시작HHMM}-{종료HHMM}_{주제}_lane-yeon-{2|3|4}_[작업중].md`
   - 각 파일은 다음 섹션을 포함:
     - 시작
     - 진행 내역
     - 차수별 계획(최소 1차)
     - 차수 항목 내 `작업내용`, `논의 필요`, `선택지`, `추천`, `사용자 방향`
2. 필요 시 각 lane 전용 기능 백로그
   - `docs/product/backlog/{주제}-lane-yeon-{2|3|4}-YYYYMMDD.md`

## 출력 포맷 (차수 템플릿)

각 차수는 아래를 포함해야 한다.

```md
## 1차 — <주제>

### 작업내용

- ...

### 논의 필요

- ...

### 선택지

1. ...
2. ...

### 추천

- ...

### 사용자 방향

- 추천 기준
````

## 실행 규칙

- `yeon` 메인 체크아웃을 직접 수정하지 않는다.
- 기존 작업이 이미 존재하면 신규 lane 전용 파일로 분리한다.
- 하나의 작은 요청은 하나의 lane에만 적재한다(작은 작업을 3개로 쪼개지 않는다).
- 동일 lane 내 동일 주제는 파일명 중복을 피하고 `-v2`, `-v3` 식으로 버전 처리한다.
- 생성 후 다음만 응답:
  - `생성된 파일 목록`
  - `lane별 다음 실행 우선순위`

## 사용 예시

- `/make-3-agent-backlog-plan 신규기능: 타자 모듈에 오프라인 큐 상태 대시보드 추가`
  - 기본은 yeon-2 단일 생성
- `/make-3-agent-backlog-plan yeon-3: 카드룸 생성 유도 문구 개선; yeon-4: community-feed 댓글 중복 렌더 버그 수정`
  - yeon-3, yeon-4 각자 1개씩 분리 생성
