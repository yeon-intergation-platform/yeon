# Clarify — 의도 명확화

좋은 프롬프트보다 중요한 건 내 의도를 명확하게 만드는 것이다.
이 스킬은 사용자가 막연한 생각을 구체적인 작업 지시로 변환할 수 있도록 객관식 질문을 던진다.

## 사용 시점

- "뭔가 해야 하는데 어디서부터 시작할지 모르겠을 때"
- "방향은 있는데 구체적인 범위가 안 잡힐 때"
- "여러 선택지 중 하나를 골라야 하는데 기준이 불명확할 때"
- 인자로 막연한 생각을 전달할 수 있다: `/clarify 상담 기록 화면 개선하고 싶은데`

## 실행 절차

### Phase 1: 맥락 파악 (자동)

사용자가 입력한 내용(있다면)과 현재 코드베이스 상태를 기반으로:

- 현재 브랜치와 최근 작업 내역 확인
- 변경 중인 파일이 있다면 어떤 영역인지 파악
- MEMORY.md에서 관련 프로젝트 맥락 확인
- **동시 작업 인식**: 다른 에이전트가 동시에 작업 중인 브랜치나 파일이 있는지 확인한다. 요청한 작업 범위가 현재 다른 에이전트가 진행 중인 영역과 겹치면 이를 명시하고 범위 조율을 먼저 제안한다.

### Phase 2: 범위 질문 (객관식)

```
## 어떤 종류의 작업인가요?

- A. 새 기능 추가
- B. 기존 기능 개선/수정
- C. 버그 수정
- D. 리팩토링/정리
- E. 설계/아키텍처 결정
- F. 기타 (직접 입력)
```

### Phase 3: 깊이 질문 (선택에 따라 분기)

사용자의 Phase 2 답변에 따라 후속 질문을 던진다. 각 질문은 객관식이다.

**A/B 선택 시 (기능 관련):**

```
## 영향 범위는?

- A. 단일 컴포넌트/페이지
- B. 여러 컴포넌트에 걸친 기능
- C. API + 프론트엔드 모두
- D. 아직 모르겠다

## 누가 사용하는 기능인가요?

- A. 최종 사용자 (UI)
- B. 개발자 (DX, 내부 도구)
- C. 둘 다
```

**C 선택 시 (버그):**

```
## 버그 상태는?

- A. 재현 방법을 알고 있다
- B. 간헐적으로 발생한다
- C. 에러 로그/메시지만 있다
- D. "뭔가 이상한데" 수준이다
```

**D 선택 시 (리팩토링):**

```
## 리팩토링 동기는?

- A. 코드가 너무 복잡해서 이해가 안 된다
- B. 새 기능 추가 전 정리가 필요하다
- C. 성능 문제가 있다
- D. 구조가 CLAUDE.md 원칙에 맞지 않는다
```

**E 선택 시 (설계):**

```
## 결정해야 할 것은?

- A. 데이터 모델 / 스키마
- B. API 설계 (엔드포인트, 계약)
- C. 컴포넌트 구조 / 상태 관리
- D. workspace 간 경계 / 의존 방향
- E. 기타 (직접 입력)
```

### Phase 4: 정리 및 확인

모든 답변을 종합하여 구체적인 작업 명세를 제시한다.

```
## 정리된 의도

**작업**: [한 문장 요약]
**범위**: [영향받는 파일/영역]
**제약**: [지켜야 할 원칙]
**완료 기준**: [언제 끝난 것인지]

이대로 진행할까요? 수정할 부분이 있으면 말씀해주세요.
```

## 핵심 원칙

- 질문은 최대 3라운드. 그 이상은 오히려 방해가 된다.
- 각 라운드의 질문은 4~5개 선택지 이내로 유지한다.
- "기타/직접 입력" 옵션을 항상 포함하여 객관식에 갇히지 않게 한다.
- 이미 명확한 의도를 가진 요청에는 이 스킬을 쓰지 않는다. 바로 실행이 낫다.
- Phase 4의 정리 결과가 사용자의 의도와 맞는지 반드시 확인받는다.
