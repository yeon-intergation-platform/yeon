# Code Review

## 리뷰 목표

"어떤 리뷰가 와도 상태 전이, source of truth, 실패 경계, 사용자 영향까지 근거로 설명하고 방어할 수 있는 코드"를 만드는 것.

## 기본 수량 목표

- **critical**: 3개 이상
- **major**: 3개 이상
- **minor**: 3개 이상

실제 이슈가 이 수에 도달할 때까지 **범위를 넓혀 탐색**한다.

### Escape clause — 실제 이슈가 수량에 못 미칠 때

**억지로 채우지 않는다.** 범위를 넓혀 전수조사한 뒤에도 실제 이슈가 하한에 미달하면, 그 사실을 **근거와 함께** 보고하고 끝낸다. 0개 리포트도 정당한 결과다.

- 형식 예시:
  > 탐색 범위: `apps/web/src/features/** (X 파일) / packages/** (Y 파일) / 설정·빌드 파이프라인 포함`
  > 전수조사 완료 후 실제 발견: critical N개 / major N개 / minor N개
  > 미발견 severity 는 탐색 결과 실질 이슈가 없었음을 의미하며, 허위 이슈로 채우지 않았음을 명시한다.
- 허위·억지 이슈는 리뷰 신뢰도를 떨어뜨리고 후속 수정 우선순위를 왜곡한다.
- "수량 하한 = 억지 채움"이 아니라 "수량 하한 = 탐색 범위 확장 의무"다. 범위 확장 후에도 없으면 없다고 말한다.

## 동시 작업 인식

이 저장소에는 여러 에이전트가 동시에 작업 중일 수 있다. 리뷰 전 반드시 아래를 전제로 삼는다.

- 코드가 미완성처럼 보이거나 임시 구현으로 보여도 현재 다른 에이전트가 작업 중인 WIP일 수 있다.
- `git status`, `git log`, 브랜치 목록을 확인하여 현재 어떤 변경이 staged/unstaged 상태인지 파악한다.
- 리뷰 대상이 "커밋된 코드"인지 "아직 작업 중인 코드"인지 먼저 구분한다.
- WIP로 의심되는 코드를 이슈로 올릴 때는 반드시 "(WIP 가능성 있음 — 완료 후 재확인 필요)"를 명시한다.
- WIP 코드에 대한 리뷰는 구조적 방향성 피드백에 집중하고, 미완성 자체를 critical로 올리지 않는다.

## 사고 원칙

리뷰 전 과정에 아래 사고 방식을 적극 적용한다.

- **논리적 사고**: 코드 흐름을 전제 → 중간 단계 → 결과로 엄밀하게 추론한다. "이럴 것 같다"가 아니라 "이 조건이 성립하면 반드시 이 결과가 나온다"를 증명한다.
- **문제해결적 사고**: 문제가 발생했을 때 어떤 사용자가 어떤 행동을 해야 재현되는지 구체적 시나리오를 만들어 검증한다.
- **복잡한 논리 흐름 사고**: 비동기 체인, 상태 전이, 이벤트 순서 등 여러 단계가 얽힌 흐름을 처음부터 끝까지 머릿속에서 실행해본다.
- **시나리오적 사고**: 정상 경로뿐 아니라 "느린 네트워크", "빠른 연속 클릭", "탭 전환", "페이지 이탈", "빈 데이터", "대용량 데이터" 등 극단 시나리오를 의도적으로 구성한다.
- **보수적 사고**: "이 코드가 맞을 것"이라는 낙관적 전제를 배제하고, 모든 외부 입력과 비동기 타이밍을 적대적으로 가정한다.
- **걱정이 많은 사고**: 사소해 보이는 코드도 "이게 실패하면 무슨 일이 벌어지지?", "이 경로가 두 번 실행되면?", "이 값이 null이면?"을 끊임없이 질문하며 탐색한다.

## 리뷰 절차

### Phase 1: Source of Truth 식별

1. 변경된 코드에서 원본 상태(source of truth)를 식별한다.
2. 파생 상태와 원본 상태를 구분한다.
3. 원본이 변경될 때 파생 상태가 함께 갱신/폐기되는지 확인한다.

### Phase 2: 상태 정합성 검증

1. `set`/`save`/`write`/`add` → 대응하는 `delete`/`clear`/`remove`/`reset` 확인
2. 캐시, 메모이즈, 폼 상태, 저장소 → 원본 변경과의 동기화 확인
3. 비동기 로직 → 순서 뒤집힘, 중복 실행, race condition 확인
4. 서버/클라이언트/SSR/route handler → 같은 개념의 규칙 일치 여부 확인

### Phase 3: 실패 경계 검증

1. 값 없음, decode 실패, null, undefined, empty 처리
2. 이전 값이 남아있을 가능성
3. 중복 호출, 재시도, 부분 실패
4. 로그아웃, 만료, 예외 시 정리 여부

### Phase 4: 경계 위반 확인

1. `apps/web/src/components` → `features`/`app` 의존 여부
2. `packages/*` → `apps/*` import 여부
3. `apps/mobile` → `apps/web/src/server` import 여부
4. 웹 전용 Server Actions로 공용 기능을 닫았는지

### Phase 5: 구현 품질

1. TypeScript `any`, `as` 단언 남용
2. CSS Modules 전역 셀렉터 단독 사용
3. Tailwind 동적 클래스 생성
4. 존재하지 않는 API/export 추측 생성
5. 로그·오류 메시지 한국어 여부

## 리뷰 코멘트 형식

각 이슈는 아래 형식으로 작성한다.

```
### [severity] 제목

- **위치**: 파일:라인
- **불변식 위반**: 어떤 불변식이 깨지는가
- **사용자 영향**: 어떤 사용자 영향이 생기는가
- **원인**: 왜 그런 버그가 생기는가
- **수정 제안**: 구체적인 수정 방향
```

## 리뷰 체크리스트 (최종)

- [ ] 이 값의 원본은 무엇인가
- [ ] 실패하면 이전 상태가 남는가
- [ ] 로그아웃, 만료, 예외 시 정리되는가
- [ ] 서버와 클라이언트 규칙이 같은가
- [ ] 여러 번 실행해도 상태가 꼬이지 않는가
- [ ] import 경계를 위반하지 않는가
- [ ] API 계약과 실제 구현이 일치하는가
